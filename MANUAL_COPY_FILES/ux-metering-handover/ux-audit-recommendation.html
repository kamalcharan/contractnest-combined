ContractNest · UX Audit & Recommendation

One spine, three skins: a north-star UX for Catalog Studio, Contracts, and Templates

A code-level audit of /catalog-studio/configure, the contract creation wizard, and both template creation flows — followed by a concrete proposal for the ultimate UX and a phased roadmap to get there.

Scope: contractnest-ui Method: full source walkthrough with file:line evidence July 2026
The verdict Good bones, fragmented delivery

Individually, each flow shows real product thinking — per-type wizard steps, server-side contract drafts, a shared Floating Action Island, VaNi AI entry points. But the three flows were built as parallel implementations that never converged: four wizard frameworks, two toast systems, two template-authoring paradigms, two block libraries, and roughly 7,000 lines of orphaned dead code sitting beside the live code. The user pays for this in the three ways that matter most in a long-form authoring product: lost work (no refresh protection, a Save Draft button that does nothing), silent walls (disabled buttons with no explanation), and broken bridges (the "Use template" button drops the template on the floor).

Part 1 · Where things stand

The current landscape: four wizards, one product

The first structural surprise: /catalog-studio/configure is not a wizard at all. It is a three-panel Block Library (category rail · card grid · slide-in editor); the actual block wizard lives behind blocks/new and blocks/:id/edit as a separate 442-line component with per-type step sequences (2–6 steps depending on block type). The contract wizard is a 2,906-line modal component with 12 steps (its own header comment still says 8), reused in template mode with 9 steps and RFQ mode with 6. Templates can additionally be authored in a completely different drag-and-drop canvas, and admins get a third, separate 9-step global designer.

What's hurting users today

Findings are ordered by user impact. Every claim is anchored to source; the evidence appendix at the end lists exact locations.

Trust & data safety

Critical"Save Draft" in the block wizard does nothing

The button is prominently rendered in the wizard footer, but its handler is a console.log. A user who clicks it, sees no error, and closes the tab has lost everything — while believing it was saved. This is the single most trust-damaging pattern in the product.

BlockWizardContent.tsx:278–281 · WizardFooter.tsx:155–168
CriticalRefresh or mis-navigation silently destroys work in three of four flows

There is no beforeunload handler anywhere in the app and no router blocker. The block wizard has zero persistence. The contract wizard autosaves only when you press Continue after the details step — steps 0–3 (path, nomenclature, acceptance, counterparty) are never saved. The admin global designer persists nothing at all across its 9 steps.

ContractWizard/index.tsx:805, 1165 · global-designer/index.tsx:80 · grep "beforeunload" = 0 hits
Critical"Use this template" is a broken bridge

The templates list navigates to /contracts?action=create&template=<id>, but the contracts hub never reads either query param. The user lands on a plain list page, template silently discarded. Several other links on the templates pages point to routes that don't exist and fall through to the 404 page.

Guidance & feedback

MajorValidation blocks silently instead of guiding

The contract wizard's Continue button simply greys out with no explanation of what's missing. The block wizard validates only on Next and surfaces errors as a summary in the footer — no inline field errors, no red borders, no scroll-to-field, so on a long step the user can't see why Continue "did nothing". Worst case: the global designer validates the template name at final save with a destructive toast, then force-jumps the user back 8 steps.

Same artifact, two entirely different UIs — users can't build a stable mental model, and every improvement must be built twice.

Part 3 · Recommendation

The ultimate UX: one spine, three skins

The insight the audit forces: ContractNest doesn't have a catalog problem, a contract problem, and a template problem. It has one problem — long-form authoring — expressed three times. Blocks, contracts, and templates are all "build a structured document step by step, with money attached". The ultimate UX is therefore not three redesigns; it is one authoring spine that all three flows wear as skins, so every guarantee (never lose work, always know why you're blocked, always edit from review) is built once and holds everywhere.

The one-sentence north star: a user can start any authoring flow, walk away mid-step, come back on any device, resume exactly where they left off, always see why they can't proceed, and always jump from the review straight to the field that needs fixing.

Pillar 1

WizardShell — a single authoring framework

Extract one shell component that owns everything the four wizards currently reimplement, and let each flow supply only a step registry and step bodies:

  • Never lose work. Debounced autosave from the first keystroke (server draft where an API exists, localStorage fallback where it doesn't), plus a router blocker and beforeunload guard. The draft pill ("Saving… / Saved") the contract wizard already has becomes universal.
  • Resumable by URL. Wizard state keyed by draft id and step in the route (/contracts/new/:draftId/:step). Refresh, share, or reopen on another device and you land exactly where you were. This also kills the modal-wizard pattern's biggest weakness — the browser back button currently exits the whole wizard.
  • One validation contract. Each step exports validate() → field-level errors. The shell renders inline errors at the field, an error summary that scrolls-to-field on click, and — crucially — never a silently disabled button: Continue stays enabled and, when pressed with errors, focuses the first problem. A short hint under the button ("Add a contract name to continue") replaces the mystery grey-out.
  • One progress model. A single stepper (side rail on desktop, collapsed top bar on mobile): completed steps always clickable, future steps visibly locked with a reason. Delete the second dot-row; keep the Floating Action Island as the action bar only (Back / Continue / draft status).
  • Review as a contract, not a page. Every flow ends in a generated review composed of per-step summary cards, each with an Edit link that deep-links to that step and returns to review on save. This single pattern fixes the block wizard's missing review, the contract wizard's 1,659-line dead-end, and the global designer's late-toast failure at once.
  • One kit. One toast provider, one loader (VaNi-branded for full-screen, one spinner for inline), one confirm dialog, one Button. Retire window.confirm, inline-style hover hacks, and bare "Loading..." text in the same sweep.
Pillar 2

Contract wizard — 12 steps become 4 phases, and templates become "review-first"

  • Group, don't just count. Keep the 12 granular steps but present them as four named phases: Setup (path · nomenclature · counterparty), Terms (acceptance · details · billing cycle), Scope (assets · service blocks · billing view), Finalize (evidence · events · review). The stepper shows phases; steps become sub-progress within a phase. Twelve dots feels like a tax form; four chapters feels like a guided build.
  • Review-first from template. Today "From Template" is marked recommended but still walks the full wizard. Invert it: choosing a template lands the user directly on a pre-filled review screen, with every section marked "from template" and editable in place. They only touch what differs — counterparty, dates, price. This turns the 80% case from a 12-step march into a 2-minute confirm, and it's the feature that makes templates actually worth authoring.
  • Split the details cliff. ContractDetailsStep's ~28 inputs collapse into essentials + progressive disclosure ("Advanced terms" accordion for grace periods, EMI, tax nuances), with smart defaults from nomenclature and tenant settings.
  • Fix the bridge. "Use template" from the templates list must open the wizard with the template applied — one consistent handoff mechanism (router state or a real query-param reader), used by every entry point, with a test that fails if it breaks again.
Pillar 3

Templates — one paradigm, with the canvas demoted to a step

  • Converge on the wizard as the single template-authoring flow (tenant and admin skins of the same shell; the admin skin adds industries, recipe slots, and publish states as extra steps).
  • Keep the drag-and-drop canvas — as the "Arrange blocks" step, not a rival full-page paradigm. Its strength (spatial composition of blocks) is a step-sized job; its weakness (no guidance, no validation, no draft) disappears once it lives inside the shell.
  • Show lifecycle honestly. Only signed-off templates are usable in contracts today, but nothing on the templates list communicates that. Add lifecycle badges (Draft / In review / Published) with the publish action inline, so "why can't I use my template?" answers itself.
Pillar 4

Catalog Studio — a store-manager's library, with an honest wizard

  • The library concept is right — commit to it. Delete the "classic" 1,094-line blocks page, repoint the equipment page's link, and make configure the only library. Give it a real zero-state ("Your catalog is empty — start with a Service block" + template gallery), skeleton cards instead of "Loading blocks...", and the same confirm dialog as everywhere else.
  • Make Save Draft real or remove the button today. Under the shell, block drafts persist like contract drafts, and the button disappears entirely because autosave subsumes it.
  • One step registry. Collapse the two duplicated step-definition files into one; move each step's validation into the step module beside its UI; add the missing review step ("Here's your Service block: 3 resources, ₹12,000/quarter, 2 business rules — Save?").
  • Fix the numbering. In edit mode, hide the type step from the count instead of starting at a visible "step 2".
Pillar 5

VaNi — one assistant, honestly present everywhere

  • One assist surface across all three flows: the same "Draft with VaNi" affordance in the block wizard ("describe the service, I'll draft the block"), the contract wizard (already real), and template authoring — same placement, same visual language, same apply/review interaction.
  • Never fake it. Feature-flag the admin recipe generator off until the backend exists; fake streaming over a stub costs more trust than the feature's absence.
  • VaNi as the guide, not just the generator: the shell's validation hints and empty states are natural VaNi moments ("Most electronics-service tenants add a preventive-maintenance block here").
Part 4 · Roadmap

Sequenced by user pain, not by architecture

Phase 0 is deliberately small and shippable in days — it removes every active trap without touching architecture. The shell lands in Phase 1 and everything after is a re-skin onto it.

PhaseThemeContentsEffort
0 Defuse the traps Remove/wire the no-op Save Draft · add beforeunload + router guards to all wizards · fix "Use template" handoff and the 404 links · disable save buttons while saving · scroll-to-error on blocked Continue · flag off the stub AI panel ~1–2 wks
1 Build the spine Extract WizardShell (stepper, autosave/resume by URL, validation contract, review-with-edit, one toast/loader/confirm) · migrate the block wizard first as the smallest proof · delete dead trees and the classic library ~3–4 wks
2 Contract flow Migrate ContractWizard onto the shell · 4-phase grouping · split the details cliff with progressive disclosure · review-first template path · full-route wizard (goodbye modal), browser back works ~4–5 wks
3 Templates & VaNi Converge tenant + admin template authoring onto the shell · canvas becomes the Arrange step · lifecycle badges + inline publish · unified VaNi assist surface (real backends only) · mobile-responsive stepper ~4 wks

Success measures worth instrumenting from Phase 0: wizard abandonment rate per step, drafts resumed vs. abandoned, time-to-contract from template (target: under 3 minutes with review-first), and support tickets mentioning "lost my work".
