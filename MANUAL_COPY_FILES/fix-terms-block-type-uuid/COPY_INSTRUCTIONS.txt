Branch: fix-terms-block-type-uuid
(Onboarding blocker — sharma@s.com got stuck at the Terms & Conditions
step with "invalid input syntax for type uuid: text")

WHAT WAS WRONG: TermsConditionsStep.tsx creates the tenant's singleton
Terms & Conditions block by calling blockToCreateData({ categoryId:
'text', ... }). That adapter sends the literal string 'text' as the
block's type/category but never sets block_type_id — the actual DB
column, which is uuid. Nothing downstream (contractnest-api's
catBlocksService, the cat-blocks edge function) resolves the friendly
string into its real UUID; they pass it straight through, and Postgres
rejects the insert. This made EVERY first-time tenant fail at this step
(only updates to an already-existing block were safe, because the
update path strips non-UUID block_type_id before sending).

THE FIX: matches an existing, established convention already used
elsewhere in this codebase (Screen8APricingStep.tsx,
ktCatBlockMapperService.ts) — hardcode the platform's permanent LOV
UUID as a constant instead of relying on server-side resolution that
doesn't exist. Confirmed the real UUID via m_category_details
(cat_block_type / sub_cat_name = 'text'): db4bf715-dc1a-46f6-94a9-
e64429137d3f.

Files Changed:

  contractnest-ui (1 file):
  - src/pages/onboarding/steps/TermsConditionsStep.tsx
      Added BLOCK_TYPE_TEXT constant. Both blockToCreateData(...) and
      blockToUpdateData(...) calls now pass categoryId: BLOCK_TYPE_TEXT
      instead of the literal string 'text'. The read-side "find existing
      T&C block" matching logic (blocks.find(b => b.categoryId ===
      'text')) is untouched — it correctly compares against the
      friendly name resolved on read via block_type_name, not the UUID.

Submodules Affected: contractnest-ui only
