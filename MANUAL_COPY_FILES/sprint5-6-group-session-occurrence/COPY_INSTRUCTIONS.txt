CONTEXT (architecture debate, resolved 19 Jul 2026)
────────────────────────────────────────────────────
BBB is the base tenant for Sprint 5 (only tenant with Group Session
catalog blocks: "Bi Weekly Meetings", "Saturday Cadence", both
audience=group, 14-day cadence). Live data check found BBB models its
~50 members as SEPARATE contracts (10 seeded so far, CN-1061..CN-1070),
each its own AR — NOT one contract with an embedded roster. That means a
"Saturday" occurrence is 50 independent per-contract service-event rows
that happen to land on the same date via the shared cadence anchor, with
nothing tying them together today.

Debated and settled: t_appointments cannot represent this (contract_id
and event_id are both NOT NULL, singular — one appointment = one
contract's one event, confirmed via schema). Retrofitting it to span
many contracts would break the existing 1:1 appointment kanban and
conflates two different jobs: "negotiate a date with one buyer"
(appointments, real) vs. "assign a host to a cadence-fixed occurrence
many buyers share" (this new table). Owner confirmed this framing:
"a Group Session is headed by someone... he will have an appointment
against 'Group Session' in the cadence" — i.e. ONE assignment per
occurrence, not one per attendee.

Also settled: t_contract_events IS already the unified "cadence" layer
VaNi (future Virtual Employee) should read/act on — it already spans
service/billing/spare_part with per-type configurable status
(m_event_status_config). Appointments and this new occurrence-assignment
entity are both subordinate HELPERS that specific event types use to
advance their own status — not peers of "Cadence," and not something to
rebuild as a new top-level object.

WHAT THIS BATCH DOES (Phase 1 of the combined Sprint 5+6 POA)
────────────────────────────────────────────────────
New table: t_session_occurrence_assignments
  (tenant_id, block_id [TEXT, matches t_contract_events.block_id],
   scheduled_date) -> status, assigned_to, assigned_to_name,
   roster_submission_id (set by Phase 2, not yet built), version,
   is_live/is_active, audit columns. Unique on
   (tenant_id, block_id, scheduled_date, is_live) WHERE is_active.

New RPC: assign_session_occurrence(tenant_id, block_id, scheduled_date,
  assigned_to, assigned_to_name, is_live, changed_by, changed_by_name)
  - Verifies at least one member event exists for the occurrence (else
    NOT_FOUND — cannot assign a session that doesn't exist).
  - Upserts the occurrence's own assignment row.
  - Fans assigned_to/assigned_to_name into EVERY member-contract's
    t_contract_events row sharing (tenant_id, block_id, scheduled_date,
    event_type='service') — ONE write action, not fifty.
  - Writes one t_audit_log row (contract_id NULL — spans no single
    contract) recording the fanout_count achieved.
  - Returns member_count + fanout_count so the caller can see the
    fan-out actually happened.

New RPC: get_session_occurrences(tenant_id, is_live, date_from, date_to)
  - Lists occurrences with a live member_count and current assignment
    state, scoped to the tenant's session-category catalog blocks via a
    plain TEXT equality join (deliberately NOT a uuid cast on
    e.block_id — FlyBy service blocks carry ids like
    "flyby-service-<timestamp>", and casting those to uuid would throw
    for every tenant that has any; a text-equality join just doesn't
    match them, which is correct and silent).
  - Powers the future cockpit consolidation view (Phase 3 — "one row per
    occurrence instead of 50" — not built in this batch).
