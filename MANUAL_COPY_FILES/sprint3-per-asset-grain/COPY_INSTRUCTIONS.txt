WHAT THIS BUILDS — Sprint 3: Contract View, per-asset grain
───────────────────────────────────────────────────
1. Per-asset progress on event cards. Each service event's card now shows
   a collapsed "n/m assets proven" chip; expands to a list of every
   covered asset with its own status (open/assigned/in_progress/proven/
   awaiting asset), assignee, and name. Works identically for seller
   (Operations tab) and buyer (My Services tab) — both already share the
   same components.
2. Equipment tab now distinguishes real vs. placeholder assets:
   - Placeholder cards get a "To be attached" badge.
   - When a contract has multiple items of the same type (e.g. 3 HVAC
     units), each card shows "N of M" so they're no longer visually
     identical.
   - Placeholder cards get a new "Attach Asset" button — opens the
     registry picker pre-filtered to that exact asset type, and replaces
     that specific placeholder in place (rather than appending a new
     item) when you pick a real asset.
3. Dead-surface cleanup: removed the unreachable "Communication" tab case
   (confirmed dead — not wired to any nav button) and its now-unused
   PlaceholderTab component; removed leftover debug console.log calls in
   the contract detail page. "Generate Invoice" (previously a dead button
   with no click handler, in both Operations and My Services/Tasks) is now
   disabled with a "arrives in Sprint 4" tooltip instead of silently doing
   nothing.
4. Same coverage_types-based Equipment/Facility classification fix from
   the previous batch, applied to the "which Add button to show" logic
   too (equipTabMode) — it had the identical gap as the tab-label fix.

BACKEND WORK BEHIND #1 AND #2
───────────────────────────────────────────────────
- New read endpoint: GET /api/contracts/:id/event-assets — returns
  t_contract_event_assets rows grouped by event_id. Routed through
  contractnest-api (routes/controller/service, thin proxy) to a new
  handler in the "contracts" edge function.
- "Attach Asset" wiring: the buyer/seller add-equipment RPCs already
  supported replacing a placeholder in-place (p_replaces_item_id, built in
  an earlier session) and already correctly unlock the right
  t_contract_event_assets rows — but the edge function's HTTP handlers
  never read/forwarded that parameter from the request body. Fixed in
  both handlers; threaded through the UI mutation hooks
  (useBuyerAddEquipment/useSellerAddEquipment) with a new optional
  replacesItemId param.
