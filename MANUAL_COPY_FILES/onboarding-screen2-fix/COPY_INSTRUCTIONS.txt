WHY THIS FIX EXISTS
─────────────────────────────────────────────────────
1. FIELD COVERAGE: The original BusinessDetailsStep manually constructed
   the API payload with only 6 fields and missed many required fields.
   This rewrite uses useTenantProfile() hook so formData IS the payload.

2. 404 on PUT: Both BusinessDetailsStep and PersonaSelectionStep were
   calling api.put() which hits the edge function's PUT handler.
   PUT does .update().select() — returns empty array and 404 when no
   profile row exists yet (new registration).

   ROOT CAUSE: The edge function's POST handler does UPSERT
   (onConflict: 'tenant_id'). POST = create OR update.
   PUT = update only, fails with 404 if row does not exist.

   FIX: Both steps now use api.post() — UPSERT always works
   regardless of whether the profile exists.

3. business_name required: The edge function requires business_name
   in both POST and PUT. PersonaSelectionStep was only sending
   { business_type_id }. Fixed by including business_name: currentTenant.name.

─────────────────────────────────────────────────────
HOW FIELD MAPPING WORKS (to address the concern)
─────────────────────────────────────────────────────
1. useTenantProfile({ isOnboarding: true }) gives us:
     formData  — full TenantProfile object (all fields)
     updateField(field, value)  — updates any field by name

2. OrganizationDetailsForm receives:
     formData={formData}          — reads all fields
     onUpdate={updateField}       — writes any field back

3. On Continue, the payload is:
     await api.put(PROFILE, { ...formData, logo_url })

   That spread sends EVERY field in TenantProfile to the API.
   There is NO manual payload construction — the interface is
   the single source of truth.
