WHAT WAS WRONG (confirmed against real data, not guessed)
────────────────────
get_appointments_list — the RPC behind /ops/appointments — INNER
JOINed t_contract_events and t_contracts, both requiring non-null
event_id/contract_id. Group Session chair appointments (created by
gs_schedule_assign when you assign a chair) deliberately have those
two columns NULL — they link via group_session_occurrence_id instead.
Result: for BBB specifically, 0 regular appointments + 18 real,
active, accepted Group Session appointments existed, and the kanban
showed "No appointments yet" — all 18 were silently excluded. Verified
directly: before the fix, get_appointments_list returned 0 rows for
BBB; after, it returns all 18, and the untouched regular-appointment
path for another tenant still returns exactly the same 17 it did
before (unaffected).

WHAT CHANGED
────────────────────
get_appointments_list now UNIONs two branches: the existing contract/
event path (byte-for-byte unchanged) plus a new group-session path
that joins t_group_session_schedule + m_cat_blocks instead. Since a
chair appointment has no single "buyer," a few fields are repurposed
so the existing card UI (which already gracefully hides fields that
are null) shows something sensible:
  buyer_name    -> the session block's name (e.g. "Saturday Cadence") — card title
  block_name    -> literal "Group Session" — subtitle, mirrors "Service visit"
  contract_name -> "Session #<seq>"
  event_date    -> the occurrence's date
  buyer_phone / buyer_email / contract_number / task_id -> NULL (simply don't render)

Also verified: update_appointment (accept/decline/reschedule/complete)
already works correctly on these rows with zero changes needed — its
event-sync block only fires when an event_id resolves, which it never
does for a null event_id, so it safely no-ops. Confirmed this live: a
real appointment was round-tripped through an update call and back,
zero side effects, zero stray audit rows.
