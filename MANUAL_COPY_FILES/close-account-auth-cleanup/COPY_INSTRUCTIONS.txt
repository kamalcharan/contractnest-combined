BATCH: close-account-auth-cleanup
SUBMODULE(S) AFFECTED: contractnest-edge

WHAT THIS BATCH IS
-------------------
Fixes the admin "Close Account" flow (/admin/subscription-management) so it
also deletes the now-orphaned auth.users record for any user left with zero
remaining tenant memberships after a tenant close. Previously the
admin_close_tenant_account() RPC computed orphan_user_ids correctly but only
returned them for manual deletion via the Supabase Dashboard — auth.users
was never actually touched.

Fix is entirely in the edge function (not the RPC): after a successful
close-account RPC call, the handler loops orphan_user_ids and calls
supabase.auth.admin.deleteUser() for each (best-effort, one failure doesn't
block the rest — same swallow-and-report style as the RPC itself). This
matches the existing auth.admin.deleteUser() usage already in this same
function's createTenant.ts rollback path.

Verified before writing this fix (via direct Supabase queries):
  - t_user_tenant_roles.user_tenant_id -> t_user_tenants.id is ON DELETE
    CASCADE, so no FK blocks the RPC's existing `DELETE FROM t_user_tenants`
    step (an earlier hypothesis that this was broken turned out to be wrong).
  - No table in the public schema has a declared FK into auth.users, so
    calling auth.admin.deleteUser() for a genuinely-orphaned user will not hit an
    unexpected FK violation from app tables. (The RPC already nulls/deletes
    the app-level references it knows about -- t_audit_logs.user_id,
    t_contacts.auth_user_id, t_user_auth_methods, t_user_profiles -- before
    returning orphan_user_ids.)
