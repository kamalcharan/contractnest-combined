WHAT THIS DOES
  The onboarding "What do you work with?" step (ResourcePickStep, the Data-Setup
  phase) used to order resource templates by a STATIC, global signal
  (is_recommended / popularity_score — identical for every tenant in an industry).
  This overlays a PER-TENANT ICP relevance ranking on the SAME list:
    • rank-to-top — the most on-profile sub-categories AND rows float to the top
    • a "★ For you" badge on templates that matched the tenant's real ICP
  Decision locked with you: RANK-TO-TOP, KEEP SELECT-ALL — the existing auto-tick
  (all KT-ready equipment + all facilities) is UNCHANGED. Nothing that ticks today
  gets un-ticked. ICP only reorders + badges.

  Ranking signal (same brain as the smart chips, so near/similar profiles get
  similar recommendations):
    • Smart Profile vocabulary (approved keywords + semantic-cluster terms)
    • ICP archetype keywords (m_icp_archetypes matched by industry / persona)
    • Embedding-neighbour terms (icp_similar_tenants → their cluster vocabulary)
    • The tenant's own materialized resources (re-seed / settings case)
    • Industry overlap + static signal as stable tiebreakers

  COLD-START SAFE: no ICP signal (blank/new tenant, or no SUPABASE_SERVICE_ROLE_KEY)
  → the endpoint returns {} and the step keeps its EXISTING static order. Zero
  regression. The ranking call also never blocks the step: the picker renders
  immediately in static order, then rank-to-top applies the moment ranking arrives.

NEW ENDPOINT

  → { success: true, data: { [resource_template_id]: { score, forYou } } }
  Read-only, tenant-scoped, degrades to { data: {} } on any error (never fails the step).

ENV: needs SUPABASE_SERVICE_ROLE_KEY for the ICP read (same as the chips). Without
     it → empty ranking → static order (no crash).
