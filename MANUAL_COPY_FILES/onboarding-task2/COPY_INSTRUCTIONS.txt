─────────────────────────────────────────────────────
WHY THE RPC APPROACH
─────────────────────────────────────────────────────
The API server only has the anon key (SUPABASE_KEY). Previous approaches failed:

  ✗ Direct Supabase insert — RLS car_tenant_isolation blocks anon key
    (current_setting('app.tenant_id') not set server-side)
  ✗ Edge function with user JWT — Supabase rejects expired user JWTs
    at infrastructure level before function code runs
  ✗ Edge function with anon key — Supabase edge function infrastructure
    also rejects the anon key as a Bearer token

  ✓ SECURITY DEFINER PostgreSQL function via supabase.rpc()
    → Anon key is accepted for RPC calls (PostgREST validates it, not edge infra)
    → SECURITY DEFINER makes the function run as the DB owner
    → DB owner bypasses RLS — inserts succeed without service_role key
    → Pattern is safe: function validates tenant_id from parameter, not JWT

─────────────────────────────────────────────────────
HOW IT WORKS
─────────────────────────────────────────────────────
POST /api/seeds/tenant/industry-confirmed
  Headers: Authorization: Bearer <token>, x-tenant-id: <uuid>
  Body:    { "industryId": "lifts_elevators", "businessType": "buyer" }

  Step 1  Idempotency guard — checks existing data before each operation
  Step 2  Clones global KT blocks (m_cat_blocks WHERE tenant_id IS NULL
          AND tags contains industryId) → tenant-scoped copies with
          is_live=false, is_seed=true
  Step 3  Buyer/both only: calls supabase.rpc('seed_onboarding_facility_nodes')
          which inserts one t_client_asset_registry placeholder per hierarchy
          level (ownership_type='self', is_live=false, is_placeholder=true)
  Step 4  Seeds sequences for live + test environments (non-fatal if skipped)
