WHAT CHANGED
═══════════════════════════════════════════════════
contractnest-api/src/services/contractComposerService.ts
  - assembleDraft / assembleFromTemplate accept an optional
    cadenceSelections param: {block_id, cycle}[]. When a candidate block
    has a cadencePricing rate card and the requested cycle is one of its
    enabled, priced rates, that cycle (and its rate) is used instead of
    the template's saved cycle. Rate always comes from the block's own
    stored rate card — the client only ever sends a cycle name, never
    an amount (same security model as cadence-acceptance.ts).
  - buildAssetPrefill now generates placeholder equipment_details rows
    (asset_registry_id: null, specifications: {placeholder: true,
    coverage_resource_id}) for any coverage type with no attached real
    asset — matching the classic wizard's AssetSelectionStep.tsx shape.
    Previously only the allowBuyerToAdd flag was set with no actual
    placeholder row, so downstream per-asset tracking
    (t_contract_event_assets, from the sprint1b migrations) had nothing
    to react to for VaNi-composed contracts.

contractnest-api/src/controllers/vaniComposerController.ts
  - New private validCadenceSelections() validates/sanitizes the
    cadence_selections array from the request body (block_id + cycle
    strings only, capped at 50 entries).
  - Both the /assemble and /assemble-from-template handlers parse
    cadence_selections and pass it through.
