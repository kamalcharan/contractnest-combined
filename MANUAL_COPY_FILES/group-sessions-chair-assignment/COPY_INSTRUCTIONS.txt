directly in the DB: 26 real appointment rows, status='accepted',
assigned_to_name='SANJAY GHOTIKAR', all written in the same call. The bug

was write-only — it never showed what's currently assigned back, only a

component state.

WHAT THIS BATCH DOES
────────────────────
Wires the already-live gs_schedule_assign / gs_schedule_assign_default
RPCs up to the Group Sessions dashboard (/group-sessions):

1. API layer (contractnest-api):
   - POST /api/group-sessions/occurrence/:id/assign
     body: { assigned_to, assigned_to_name? } — assigned_to=null clears
     the chair (and soft-deletes the linked appointment).
   - POST /api/group-sessions/occurrences/:blockId/assign-default
     body: { assigned_to, assigned_to_name? } — sets the chair as default
     for every future, non-cancelled occurrence of that block in one call.
   - New controller methods (assignChair, assignDefaultChair) + service
     methods, following the exact existing pattern (same tenant/is_live
     header extraction, same sendSuccess/sendError shape).

2. UI layer (contractnest-ui):
   - GsOccurrenceRow gains assigned_to / assigned_to_name (already
     returned by gs_dash_occurrences per the 030/031 migrations — no
     backend gap, just wasn't typed/used yet).
   - New hooks useAssignChair() / useAssignChairDefault() in
     useGroupSessionsDashboard.ts, same react-query + toast pattern as
     every other mutation in that file.
   - New AssignChairControl (local component in the page): shows
     "Chaired by X · Change · Remove" once assigned, or an "Assign
     chair"/"Assign default chair" button otherwise. Opens the EXISTING
     common/ContactPicker.tsx inline — filtered to
     classifications: ['buyer','vendor','partner','team_member'] per
     your spec — no new picker built.
   - "Session series" card (Group view) gets the DEFAULT chair action —
     applies to every future session in one call.
   - Occurrence drill-down (click any session date) gets the PER-SESSION
     override action next to Move date / Skip, plus a small green
     "Appointment · <name> · Accepted" card once a chair is assigned —
     the visible confirmation you asked for, not a silent side-effect.
   - Picking a contact only yields an id from ContactPicker; the control
     resolves the display name via the same useContact() hook
     ContactPicker itself uses, before firing the mutation (so
     assigned_to_name is always a real name, never blank).

SCOPING NOTE
────────────────────
The Appointment Card is derived purely from the occurrence's own
assigned_to_name + date (status is always "Accepted" by design — see
031's migration notes: gs_schedule_assign always creates it in
'accepted' state). It does NOT fetch the appointment row independently.
If you later want to change a chair's appointment status from inside
this card (e.g. mark declined) rather than always showing Accepted,
that's a follow-up — the appointment itself is a real row in
t_appointments and fully reachable via the existing Appointments kanban
once that module is built out (Sprint 6, not yet started).
