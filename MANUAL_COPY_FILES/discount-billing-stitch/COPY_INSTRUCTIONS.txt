Submodules Affected: contractnest-ui (6 files), contractnest-api (2 files)

WHAT THIS FIXES
───────────────────────────────────────────────────
The contract-level discount (Billing View step) was only cosmetic in one
common case. It always displayed correctly, but:

  1. It was NEVER sent to the backend at all — create_contract_transaction /
     update_contract_transaction already accept discount_type/discount_value/
     discount_total, but the wizard's payload builder never included them.

  2. For "Per Schedule" billing (mixed cadence — each block bills on its own
     cycle; paymentMode === 'defined'), the actual generated billing events
     were computed straight from each block's undiscounted total. So a buyer
     on Per Schedule billing was invoiced the FULL undiscounted amount at
     every cycle, even though Billing View showed a discounted total.
     (Upfront and EMI billing were already correct — they use grandTotal,
     which already nets the discount.)

WHAT CHANGED
───────────────────────────────────────────────────
contractnest-ui:
  - src/components/contracts/ContractWizard/logic/mapper.ts
      Added discount_type / discount_value / discount_total to the API
      payload. Passes baseSubtotal/discountTotal into event computation.
  - src/utils/service-contracts/contractEvents.ts
      The paymentMode==='defined' branch now loads the contract discount
      pro-rata into every block's billed total before splitting it into
      per-cycle installments (same allocation Billing View already uses
      for its own tax-breakup display). Upfront/EMI branches unchanged
      (already correct).
  - src/components/contracts/ContractWizard/steps/EventsPreviewStep.tsx
    + src/components/contracts/ContractWizard/index.tsx
      The Events Preview step (what the seller sees before submitting) now
      gets baseSubtotal/discountTotal too, so the preview matches exactly
      what gets billed after submission — previously the preview would have
      shown the (wrong) undiscounted amounts for Per Schedule billing.
  - src/components/contracts/document/ContractDocument.tsx
      Same fix for the generated contract document's payment schedule
      (Schedule C), which computes off the same shared event math.
  - src/components/contracts/ContractWizard/logic/__parity__/goldens.json
      Updated golden for the one existing fixture that sets a discount
      (F1_basicContract) to include the 3 newly-emitted payload fields.
      No fixture in this harness currently exercises paymentMode:'defined',
      so this is the only line-level diff — see TESTING NOTE below.

contractnest-api:
  - src/services/contractEventsDerivationService.ts
      Server-side mirror of the exact same contractEvents.ts fix (this file
      is a "byte-identical parity" port per its own header comment — server-
      recomputed events must agree with what the buyer saw at signing).
  - src/__tests__/contractEventsDerivationParity.ts
      Mirrored the same fix into the test's inline reference copy, and
      added 2 new scenarios: discount + Per Schedule billing (the bug this
      fixes) and discount + Upfront billing (proves no double-counting).
