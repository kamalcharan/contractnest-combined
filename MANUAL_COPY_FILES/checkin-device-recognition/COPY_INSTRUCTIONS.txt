SESSION CHECK-IN — DEVICE RECOGNITION (member silent, substitute/guest one-tap confirm)
Branch folder: checkin-device-recognition
Submodules affected: contractnest-edge (already applied live), contractnest-api, contractnest-ui
═══════════════════════════════════════════════════

WHAT THIS BATCH DOES
────────────────────
Today, /checkin/:token forces every visitor to type their mobile number on
every single scan — even a member rescanning the same chapter's QR every
week on the same phone. This batch adds browser-local device recognition:

1. MEMBER rescanning the same browser → fully silent. Skips straight to
   the attendance screen, no phone entry at all.
2. SUBSTITUTE rescanning the same browser → recognised by name, with a
   one-tap "Standing in for X again today?" confirm (or "Different member
   today" if subbing for someone else this time, or "Not you?" to fall
   back to typing a number).
3. GUEST rescanning the same browser → same one-tap-confirm pattern
   ("Check in as a guest again today?").
4. "Not you? Start over" (and "Check in someone else" after a completed
   check-in) now also forgets this browser, so the next person on a
   shared device/tablet isn't silently recognised as the previous person.

IMPORTANT — this is NOT a real device ID. No web page can read a phone's
actual hardware identity (blocked deliberately on every mobile OS/browser,
whether the tab was opened via a QR scan or any other way). This is a
random token generated by the page itself and persisted in that browser's
localStorage — it only proves "same browser as last time." Clearing site
data, incognito, or switching browsers resets recognition. Real device/
login identity (opening the check-in from inside a logged-in mobile app)
was explicitly deferred — the mobile app currently has zero groundwork for
that (no deep-linking, no QR screen, no device metadata capture at all —
confirmed by reading ContractNest-Mobile's auth/login code directly).

DB (already applied live, verified against real BBB data, fully cleaned up
after testing — see migration 039 for the full record):
  - New table t_checkin_devices: one row per (tenant, block, browser
    token) → the contact it resolved to last time.
  - New RPC gs_checkin_device_lookup: resolves a browser token straight to
    a member/substitute/guest match, mirroring gs_lookup_member's phone
    lookup but keyed by the token instead.
  - gs_checkin_guest / gs_checkin_substitute / gs_submit_checkin: each
    gets a new optional p_device_token param (defaults NULL, fully
    backward compatible) that records the device→contact mapping on a
    successful check-in.

⚠ CAUGHT AND FIXED DURING THIS WORK: adding a parameter via CREATE OR
REPLACE does NOT replace a Postgres function if the parameter signature
changes — it silently creates a SECOND overload instead, and any call
omitting the new param becomes ambiguous between the two, breaking every
existing caller. Hit this live (would have broken the working check-in
flow for real BBB members) and fixed it by dropping the old signatures
before creating the new ones — migration 039 includes those DROP
FUNCTION statements so a from-scratch replay doesn't repeat the bug.
