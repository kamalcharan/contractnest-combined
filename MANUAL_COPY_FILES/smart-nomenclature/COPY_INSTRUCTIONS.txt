WHAT CHANGED
  The VaNi intent-parse LLM chooses a contract nomenclature from a CLOSED list.
  It was injecting ALL nomenclatures. Now the list is SHRUNK to the tenant's
  ICP-relevant contract types before it hits the prompt — a shorter closed list
  = faster + more accurate classification.
    • fetchNomenclatures() now carries form_settings.industries (was dropped).
    • New relevantNomenclatures(list, icp): keeps only groups that fit the tenant:
        equipment/consumable resources -> equipment_maintenance
        asset (facility) resources     -> facility_property
        service resources OR service-led (no hard resources) -> service_delivery
        flexible_hybrid                -> always kept (catch-all)
        + any nomenclature whose industries overlap the tenant's industries.
      Falls back to the FULL list when there is no ICP signal (blank/new tenant).
    • parseIntentOnly() fetches the tenant ICP (fetchTenantIcp) and injects the
      shrunk list into the prompt. IMPORTANT: the LLM's answer is still resolved
      against the FULL list (matchNomenclature), so a valid choice is never lost.

  Effect: a FOCUSED tenant (pure services / pure equipment shop) sees a tight
  list (e.g. service_delivery + flexible_hybrid only). A GENERALIST like hubb
  (equipment + asset + service) keeps all groups — no shrink, correctly.

VERIFIED
  - tsc --noEmit = 0 source errors.
  - Against live data: cat_contract_nomenclature groups are exactly
    equipment_maintenance / facility_property / service_delivery / flexible_hybrid
    — the resource-type mapping the filter uses. (The per-nomenclature industries
    vocabulary differs from m_catalog_industries, so it only ADDS relevance where
    they align, e.g. 'healthcare'; resource types are the primary signal.)

ENV: needs SUPABASE_SERVICE_ROLE_KEY (the ICP read is service-role). Without it,
     no ICP signal -> the parse list falls back to the full list (no crash).
