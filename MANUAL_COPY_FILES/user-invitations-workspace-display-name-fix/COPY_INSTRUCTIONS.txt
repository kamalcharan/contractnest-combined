BATCH: user-invitations-workspace-display-name-fix (2026-07-22)
═══════════════════════════════════════════════════
ISSUE
─────
Tenant's business name is "BBB Bhagyanagar", but the invitation email/SMS/
WhatsApp text said: "Sanjay Ghotikar has invited you to join BBB on
ContractNest."

ROOT CAUSE
──────────
createInvitation() and resendInvitation() built the invite message's
workspaceName from t_tenants.name only:
    const { data: tenant } = await supabase.from('t_tenants').select('name, ...')
    ...
    workspaceName: tenant?.name || 'Workspace'

t_tenants.name for this tenant is literally "BBB" — a short internal
name/code, NOT the business's display name. The actual display name
("BBB Bhagyanagar") lives in a separate table, t_tenant_profiles.business_name
— confirmed by querying the DB directly. This is also the field the rest of
the app already treats as canonical (dashboard header, VaNi chat, tenant
context service all read business_name), so the invitation flow was simply
missing a join to it.

FIX
───
Both createInvitation() and resendInvitation() now also fetch
t_tenant_profiles.business_name for the tenant_id, and use:
    workspaceDisplayName = tenantProfile?.business_name || tenant?.name || 'Workspace'
as the workspaceName passed into the invite message — falling back to
t_tenants.name (then a generic 'Workspace') if a tenant hasn't completed
onboarding/profile setup yet. No schema change, no other flow touched.

This batch is layered on top of the user-invitations-pagination-fix batch
(same file) — this copy contains BOTH fixes together, so copying this one
file gives you the full current state of index.ts.
