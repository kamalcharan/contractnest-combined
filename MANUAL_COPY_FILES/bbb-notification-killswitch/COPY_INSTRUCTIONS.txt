REQUEST: "for BBB tenant - disable all whatsapp and emailid going out" — BBB
is being used for live/production-adjacent testing and real messages

INVESTIGATION FINDINGS (before making any change):
  - All outbound WhatsApp and email goes through MSG91, dispatched from a
    single chokepoint: jtd-worker/index.ts's processMessage(), regardless
    of which feature enqueued the message (contract signoff, user
    invitations, payment/invoice notifications all funnel into the same
    n_jtd queue).
  - A tenant-level config table already exists for exactly this purpose —
    n_jtd_tenant_config (is_active, channels_enabled jsonb e.g.
    {email:true, whatsapp:true}, keyed by tenant_id + is_live) — and the
    3 enqueue-side callers already check it (isChannelEnabled()) before
    inserting into the queue. BUT jtd-worker itself NEVER re-checked this
    config before actually sending — so it blindly dispatched anything
    already in the queue, and any hypothetical future enqueue path that
    skipped the per-caller check would bypass the config entirely. No row
    existed for BBB, so it was implicitly enabled-by-default.
  - Checked for already-queued messages for BBB (tenant_id
    dd194710-92b4-4110-80eb-0b492a0d2c1f) before making any change: none
    pending — 19 emails already had status_code='sent' (delivered earlier
    today, cannot be recalled; flagging this so it's known, not something
    this fix can undo).

FIX — two layers, both applied live:
1. DB config (both environments): inserted n_jtd_tenant_config rows for
   BBB, is_live=true AND is_live=false, with
   channels_enabled={"email":false,"whatsapp":false,"sms":false,"push":false,"inapp":true}.
   This blocks all NEW enqueues from the 3 gated callers immediately.
2. Code fix (jtd-worker/index.ts, deployed live): added a check in
   processMessage(), right before the channel dispatch, that re-reads
   n_jtd_tenant_config for the message's tenant_id + is_live and skips
   the actual send (deletes from queue, marks failed, archives to DLQ)
   if the channel is disabled — for 'email' and 'whatsapp' specifically.
   This is a GENERAL fix (not hardcoded to BBB) — it makes the worker
   finally honor the config table that already existed for this purpose,
   for every tenant, closing the gap that layer 1 alone doesn't cover
   (already-queued messages, or any future enqueue path that skips the
   per-caller check).

the LIVE jtd-worker (before this fix) was already 1 version behind the
