BATCH: user-invitations-pagination-fix (2026-07-22)
═══════════════════════════════════════════════════
ROOT CAUSE
──────────
"Accept & Join Workspace" did nothing for kamalcharan@gmail.com. Investigation
found 3 real 400 responses (not a silent client failure): the invitation flow
called auth.admin.createUser() for an email that ALREADY had an auth.users
account (created 2025-07-15), which correctly rejected the duplicate — but the
UI/flow got there in the first place because existing-user detection wrongly
said "this is a new user".

Why detection was wrong: this project has 162 auth.users rows. Both
validateInvitation() and acceptInvitation() in this edge function had a
fallback lookup:
    supabase.auth.admin.listUsers({ page: 1, perPage: 100 })
Supabase's admin listUsers() only returns ONE page per call. Any account
past the first 100 (like this one) was silently invisible to that lookup,
so a genuinely existing user was treated as brand new and routed down the
"create account" path instead of "sign in with your existing account".

FIX
───
New shared helper `findAuthUserByEmail()` loops through listUsers() pages
(1000/page, up to 50 pages) until it finds a case-insensitive email match
or runs out of users. Both call sites (validateInvitation's fallback,
acceptInvitation's fallback) now use this helper instead of the bare
page:1 call. No other logic changed — diffed against the live pre-fix
source to confirm.

Also cleaned up as part of this: the stale duplicate auth.users record for
kamalcharan@gmail.com was deleted directly in Supabase (after nulling the
one FK reference from a closed test tenant's created_by column) so the
invitation can be retried cleanly.
