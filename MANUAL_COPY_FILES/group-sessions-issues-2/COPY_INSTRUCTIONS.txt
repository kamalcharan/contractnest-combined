ANSWERS TO YOUR 8 ITEMS
────────────────────────
1) Edge deployment — yes, all DB work below was applied directly to the
   live Supabase project and verified before this batch was written.
   There's no separate Deno edge function involved in this feature; the
   migration files are the permanent record of what's already live.

2) & 5) Wrong schedule dates / next session on a Friday — root cause
   found: the "Saturday Cadence" catalog block was missing
   anchorWeekday in its cadence config (unlike "Bi Weekly Meetings",
   which has it set correctly). Without it, the whole series was
   anchored to the linked contracts' start_date (a Wednesday) from the
   very first generation and drifted onto Thursdays — this was NEVER a
   "moving one occurrence cascades to the rest" bug (confirmed
   gs_schedule_move only ever updates the single row being moved).
   FIXED: backfilled anchorWeekday=6, deleted the wrongly-anchored
   future rows (including the 07-31 Friday you flagged as wrong), and
   regenerated. The one held/completed occurrence (07-11, real
   attendance data) was left untouched and is a genuine Saturday. Next
   session is now correctly 2026-07-25 (a Saturday). See migration 034
   for the exact repair (idempotent, scoped to that one block).

4) Freeze completed sessions — FIXED. Once an occurrence's status is
   'held' (attendance has been recorded for it), its date and chair
   are now immutable — gs_schedule_move / gs_schedule_assign refuse
   with {ok:false, reason:'occurrence_completed'}, and gs_schedule_status
   refuses to transition away from 'held'. Attendance marking
   (gs_mark_attendance) stays editable regardless, so you can still
   correct who attended after the fact. The UI now shows a
   "Completed — locked" badge on a held occurrence and hides the
   Move/Skip/Assign controls entirely (attendance marking stays
   visible and enabled).

6) Chair on all sessions with exceptions — unaffected by any of the
   above; already correct from the previous batch (default chair +
   per-occurrence override). Verified it's still working after the
   schedule repair (assigned_to lives on the same row as the date, so
   fixing dates doesn't touch it).

7) Audit trail — FIXED. Added p_changed_by/p_changed_by_name params
   (mirroring update_appointment's existing convention) to
   gs_schedule_move, gs_schedule_status, gs_schedule_assign(_default),
   and gs_mark_attendance — every real change now writes a row to the
   existing t_audit_log table (move, status change, chair
   assign/unassign, attendance mark). The API layer extracts the
   actor from req.user (same pattern as appointmentController) and
   passes it through automatically — no new UI needed to see it work,
   but there's no dedicated "view audit history" screen yet; the data
   is queryable directly for now.

8) Substitute name display — FIXED. gs_checkin_substitute already
   wrote the correct composite name into t_session_attendance.member_name
   (e.g. "Charan Kamal (substitute: Laasya Priya)"), but
   gs_occurrence_attendance — the roster the UI actually reads — was
   ignoring it and always showing the contract's own buyer_name. Fixed
   to prefer the attendance row's member_name when one exists. Verified
   directly against a real block/member: the roster now shows the
   substitute-decorated name and present=true correctly. (Kept the
   already-stored format — "Name (substitute: Sub Name)", colon — since
   that's what gs_checkin_substitute writes; let me know if you want the
   dash format you mentioned instead, that's a one-line tweak.)
