WHAT THIS IS
----------------------------------------------------------------
When a member scans the session QR, the check-in page now renders its
QUESTIONS from a Smart Forms template (configurable per tenant) instead
of hardcoded fields. The token wizard (resolve -> phone lookup -> history
-> dues) is unchanged; only the questions step became Smart-Forms-driven.

Why hybrid (not a pure Smart Form): the Smart Forms SAVE pipeline
(/api/forms/submissions -> m_form_submissions) requires a logged-in
tenant JWT + NOT-NULL tenant/template/contract keys, which a logged-out
member scanning a QR does not have. So answers keep flowing through the
token-gated gs_submit_checkin RPC. The Smart Forms RENDERER component is
reused (it is pure/auth-free); the persistence is NOT.

HOW THE FORM IS CHOSEN (per tenant, generic)
----------------------------------------------------------------
gs_checkin_form(token) serves the APPROVED Smart Forms template mapped to
the session's owner contract via m_form_template_mappings with
timing = 'during_service'. If none is mapped, it returns a built-in
attendance-only default, so check-in always works out of the box.
To customise: author a template in Smart Forms admin (status=approved) and
add a m_form_template_mappings row (contract_id = the group-session owner
contract, timing='during_service', status='active').

The well-known field id `attendance_status` (values present|apologies)
maps to attendance; all other fields are stored as `form_responses` JSONB
on t_session_attendance. Dues selection stays custom (dynamic per member).
