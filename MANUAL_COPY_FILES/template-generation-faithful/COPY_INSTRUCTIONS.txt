WHAT CHANGED (assembleFromTemplate / assembleDraft)
  When you instantiate a signed-off template (chip → "instant" path, or the
  matched-template path), the template's SETUP now actually carries through.
  Previously only `acceptance` + `nomenclature` were applied; billing/EMI/grace
  and each block's saved cycle were silently dropped (a "monthly EMI" template
  came out prepaid).

  Now the template's `settings.defaults` become the BASELINE when instantiating:
    • payment_mode = emi  -> EMI mode + emi_months from the template
    • payment_mode = defined / billing_cycle_type = mixed / blocks have cycles
        -> per-block billing, each block keeping its OWN saved cycle
    • else prepaid
    • grace_period_value/unit -> applied (as days)
  An EXPLICIT user request still wins (e.g. a chip that says "billed quarterly"
  or "on 6-month EMI" overrides the template baseline). Per-block cycle is
  carried via a new optional CandidatePayload.cycle; the LLM compose path is
  unchanged (no cycle on those candidates -> same behaviour as before).

  NOT changed (deliberate, per your architecture):
    • Buyer / dates / assets still left to the user on the review + buyer card.
    • Equipment/facility templates still expect assets to be added after (the
      "add assets" readiness prompt is correct, not a bug).
    • Contract-level tax rate ids (selected_tax_rate_ids) still deferred — per
      block taxRate is applied as before; layering contract tax rates is a
      follow-up (needs a tax-rate lookup + total reconciliation).

VERIFIED: tsc --noEmit = 0 source errors (2 unrelated tsconfig deprecation notices).
