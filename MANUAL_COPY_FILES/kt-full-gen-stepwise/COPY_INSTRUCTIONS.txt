Fix: /generate endpoint internally runs 3 focused steps
     instead of one 16k-token prompt that always fails
=====================================================

ROOT CAUSE:
  POST /api/knowledge-tree/generate (existingKT: false) was calling
  a single LLM prompt with max_tokens: 16000. It consistently hit
  max_tokens → JSON truncated → service_cycles missing → 500 error.

  This endpoint is called by the KT creation flow (any UI that creates
  a fresh KT without existingKT:true). The stepwise buttons on the
  detail page are for retry — the INITIAL generation came through here.

FIX:
  knowledgeTreeController.ts — the generate() method's existingKT:false
  branch now runs 3 focused steps internally:

    Step 1: generateVariants()     — max 2,000 tokens
    Step 2: generateSpareParts()   — max 10,000 tokens (real variant UUIDs passed)
    Step 3: generateCheckpoints()  — max 10,000 tokens (real variant UUIDs passed)

  The combined result is returned in EXACTLY the same JSON format as
  before — the UI save logic is completely unchanged.

  existingKT:true path (+ Install / + Decomm) is UNCHANGED.

WHY THIS FIXES SPARE PARTS MAPPING:
  Step 2 now receives real variant UUIDs from Step 1. The LLM uses
  them directly in spare_part_variant_map — no temp-ID resolution
  needed for variant_id entries.

FILES CHANGED:
  contractnest-api:
  1. src/controllers/knowledgeTreeController.ts  [MODIFIED]
     — generate() existingKT:false branch replaced with 3-step sequence

APPLY AFTER: kt-stepwise-gen, kt-spare-parts-fix
