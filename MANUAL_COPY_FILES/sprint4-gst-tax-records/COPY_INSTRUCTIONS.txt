WHAT THIS ADDS
───────────────────────────────────────────────────
1. A point-in-time tax_breakdown (e.g. CGST/SGST/IGST) snapshot on
   t_invoices, copied from the contract at invoice-generation time — a
   filed period's tax cannot silently change if the contract is edited
   later. [LIVE]

2. get_tenant_tax_summary RPC — month-wise (by invoice issue date,
   accrual basis): taxable value, tax invoiced, tax collected
   (approximate), invoice count, tax component split. [LIVE]

3. A "Taxes" section on /ops/finance (receivables view only) — month-wise
   table + CSV export. [DB LIVE — UI needs copying]

4. Backfill of the 27 pre-existing invoices' tax_breakdown — done, 29
   rows updated (some tenants have duplicate invoice-number sequences
   across environments), 0 total_amount drift, 1 invoice's breakdown
   stayed empty because its own contract never captured one either (a
   separate, pre-existing, out-of-scope contract-side data gap). [DONE]

5. Tax registration number (GSTIN / VAT ID / etc., via the existing
   t_tenant_profiles.gst_number column — not renamed, just relabeled
   country-aware in the UI):
   - Read-only display on Tax Settings → Tax Display panel.
   - Printed on the invoice view.
   - Printed on the wizard's Review & Send document preview.
   - Wired (defensively, no-op-safe) into the public buyer review page —
     see Finding #2 below, this one specific field won't show until a
     separate pre-existing bug is fixed.

6. "No Tax" — a third, explicit display_mode value for tenants that are
   not tax-registered, alongside the existing "Including tax" /
   "Excluding tax":
   - DB: t_tax_settings CHECK constraint widened to allow it; both RPCs
     (see Finding #1) validate/store it; no_tax forces
     default_tax_rate_id to NULL server-side regardless of what's sent.
   - UI: Tax Display panel now shows all 3 as radio options. The Taxes
     NAV on /ops/finance shows "Not tax registered" instead of an
     empty/zero table when a tenant has this set.
   - No pricing-math changes needed: contractEvents.ts's taxFactor
     already collapses to Price = Total whenever no rate applies (the
     no_tax state formalizes what already happens by omission — see the
     "Why are we saying no impact" thread in this session for the full
     reasoning). [DB LIVE — UI needs copying]

1. FIXED — tax-settings save was broken in production since ~Feb 2026.
   create_or_update_tax_settings / get_tax_settings_with_rates were
   applied directly to Supabase outside any tracked migration at some
   point before this session (5 t_tax_settings rows prove real writes
   happened, one tenant saved 7 times between Jul 2025–Feb 2026) — then

   could reconstruct what broke. Recreated both (migration 064),

   rows. NOTE: this migration was itself first applied via MCP before
   being written to this file — caught immediately and corrected so the
   exact same untracked-drift problem didn't repeat on its own fix.

2. STILL OPEN — the PUBLIC buyer review page's provider letterhead
   (logo, business name, address, phone, email, and now tax number) is a
   live no-op. The tracked migration file for validate_contract_access
   (contracts/010_public_contract_rpcs.sql) builds a full `profile`
   object; the LIVE deployed function (pulled via pg_get_functiondef,
   verified) does not — its `tenant` object only returns {id, name}.
   contractnest-ui/src/pages/contracts/review/index.tsx has been reading
   `tenant?.profile?.*` as undefined in production for all of these
   fields, not just the new tax number. This batch's edit there is
   written defensively (same undefined-safe pattern as the pre-existing
   lines) — zero regression — but won't show anything until this
   separate RPC gap is fixed. Not touched here given the scope of the
   drift and the risk of guessing what else changed without a fuller
   audit.

   mints a draft. That function has been patched across 6 separate
   migration files; reproducing it without reading its full,

4. DELIBERATELY NOT TOUCHED — a fully duplicate, parallel
