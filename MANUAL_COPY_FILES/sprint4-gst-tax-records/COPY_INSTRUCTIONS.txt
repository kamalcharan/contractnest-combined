───────────────────────────────────────────────────

   t_invoices, copied from the contract at invoice-generation time — a
   filed period's tax cannot silently change if the contract is edited
   later. [LIVE]

2. get_tenant_tax_summary RPC — month-wise (by invoice issue date,
   accrual basis): taxable value, tax invoiced, tax collected
   (approximate), invoice count, tax component split. [LIVE]

3. A "Taxes" section on /ops/finance (receivables view only) — month-wise
   table + CSV export. [DB LIVE — UI needs copying]

4. Backfill of the 27 pre-existing invoices' tax_breakdown — done, 29
   rows updated (some tenants have duplicate invoice-number sequences
   across environments), 0 total_amount drift, 1 invoice's breakdown
   stayed empty because its own contract never captured one either (a
   separate, pre-existing, out-of-scope contract-side data gap). [DONE]

5. Tax registration number (GSTIN / VAT ID / etc., via the existing
   t_tenant_profiles.gst_number column — not renamed, just relabeled
   country-aware in the UI):
   - Read-only display on Tax Settings → Tax Display panel.
   - Printed on the invoice view.
   - Printed on the wizard's Review & Send document preview.
   - Wired (defensively, no-op-safe) into the public buyer review page —
     see Finding #2 below, this one specific field won't show until a
     separate pre-existing bug is fixed.

6. "No Tax" — a third, explicit display_mode value for tenants that are
   not tax-registered, alongside the existing "Including tax" /
   "Excluding tax":
   - DB: t_tax_settings CHECK constraint widened to allow it; both RPCs
     (see Finding #1) validate/store it; no_tax forces
     default_tax_rate_id to NULL server-side regardless of what's sent.
   - UI: Tax Display panel now shows all 3 as radio options. The Taxes
     NAV on /ops/finance shows "Not tax registered" instead of an
     empty/zero table when a tenant has this set.
   - No pricing-math changes needed: contractEvents.ts's taxFactor
     already collapses to Price = Total whenever no rate applies (the
     no_tax state formalizes what already happens by omission — see the
     "Why are we saying no impact" thread in this session for the full
     reasoning). [DB LIVE — UI needs copying]

FINDINGS — all four resolved or explicitly deferred
───────────────────────────────────────────────────
1. FIXED — tax-settings save was broken in production since ~Feb 2026.
   create_or_update_tax_settings / get_tax_settings_with_rates were
   applied directly to Supabase outside any tracked migration at some
   point before this session (5 t_tax_settings rows prove real writes
   happened, one tenant saved 7 times between Jul 2025–Feb 2026) — then
   disappeared, with no migration file ever recording them, so nobody
   could reconstruct what broke. Recreated both (migration 064),
   verified live against real data with zero side effects on existing
   rows. NOTE: this migration was itself first applied via MCP before
   being written to this file — caught immediately and corrected so the
   exact same untracked-drift problem didn't repeat on its own fix.

2. FIXED — the PUBLIC buyer review page's provider letterhead (logo,
   business name, address, phone, email, tax number) was a live no-op.
   The tracked migration file for validate_contract_access
   (contracts/010_public_contract_rpcs.sql) built a full `profile`
   object; the LIVE deployed function (pulled via pg_get_functiondef,
   verified byte-for-byte before editing) didn't — its `tenant` object
   only returned {id, name}. Fixed (migration 065): added a
   t_tenant_profiles lookup and a `profile` key to the response,
   reproducing every other line of the live function verbatim. Verified:
   error path unchanged (invalid cnak still returns the same error); did
   NOT invoke the function against a real pending access record (it has
   a side effect — marks the link "viewed" — which would have corrupted
   real buyer-tracking data), instead verified the new profile lookup
   via a read-only SELECT against real data. contractnest-ui/src/pages/
   contracts/review/index.tsx's gst_number wiring (added earlier in this
   batch, written defensively for exactly this gap) now actually works —
   no UI change needed for this fix, only the RPC.

3. FIXED — the VaNi scanner's per-event DRAFT invoice path
   (run_contract_event_scanner, STEP 4) hardcoded tax_amount=0 on
   scanner-created drafts. Zero draft invoices existed in production
   when this was found (verified live), so it was latent, not an active
   undercount. Pulled the LIVE function definition first (20,782 chars,
   pg_get_functiondef) rather than trust the tracked migration file
   (operations-loop/013_stage3_scanner_v2.sql) — the live version had
   already drifted significantly beyond it (an advisory lock guard, a
   STEP 2b appointment-auto-request block, and per-tenant VaNi rule
   gating on every step, none of which are in the tracked file).
   Reproducing from the file would have silently reverted all of that.
   Fixed (migration 066): tax_amount and tax_breakdown now prorate from
   the contract's own tax configuration, proportional to the event's
   share of the contract's total_value — documented as an approximation
   (no per-line tax data exists in the schema). SELF-CAUGHT BUG: the
   first applied version used the wrong ratio for the tax_breakdown
   component split (tax_total/total_value instead of
   event.amount/total_value), double-applying the tax rate and producing
   wrong numbers. Caught via a read-only proration test against real
   contract data BEFORE calling this done — never reached a live draft
   invoice (0 exist; scanner is cron-triggered, not called on-demand).
   Corrected and re-verified. Did NOT invoke the actual scanner function
   to test (real side effects: sends real reminder emails/SMS, creates
   real appointments, marks real events overdue) — verified via
   read-only simulation only.

4. DELIBERATELY NOT TOUCHED — a fully duplicate, parallel
   useUpdateTaxDisplay implementation exists in
   contractnest-ui/src/hooks/queries/useProductMasterdata.ts (its own
   TaxSettings/TaxDisplayFormData types, its own validator, the same old
   2-value limitation). Confirmed dead code — grepped, it's imported
   nowhere else in the codebase. Not widened; no value in fixing
   something nothing calls.
