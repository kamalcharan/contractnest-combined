WHAT THIS ADDS
───────────────────────────────────────────────────
1. A point-in-time tax_breakdown (e.g. CGST/SGST/IGST) snapshot on
   t_invoices, copied from the contract at the moment an invoice is
   generated — so a filed period's tax cannot silently change if the
   contract is edited later. Mirrors the existing t_contracts.tax_breakdown
   shape.

2. A new read-only RPC get_tenant_tax_summary — month-wise (by invoice
   issue date, i.e. accrual basis): taxable value, tax invoiced, tax
   collected (approximate — no per-payment tax split exists in the schema),
   invoice count, and the tax component split.

3. A "Taxes" section on /ops/finance (receivables view only) — a month-wise
   table + CSV export, so you can answer "what tax did I invoice/collect
   in June" without touching the database directly.

4. A report-first backfill for the 27 pre-existing invoices so the Taxes
   view is historically complete, not just forward-looking.

5. Tax registration number (GSTIN / VAT ID / etc.) surfaced wherever it's
   already captured (t_tenant_profiles.gst_number — the column name stays
   as-is; only relabeled in UI as country-aware "Tax Registration Number"):
   - Read-only display on the Tax Settings page (Tax Display panel), with
     a warning if it's not set.
   - Printed on the invoice view (pages/contracts/invoice/index.tsx).
   - Printed on the wizard's Review & Send document preview
     (ContractWizard/steps/ReviewSendStep.tsx).
   - Added to the public buyer review page's document data — see the
     BLOCKED finding below; this is wired but currently a no-op live.

WHAT DID NOT NEED CHANGING (verified, not assumed)
───────────────────────────────────────────────────
- Invoice line-item display: the invoice view already itemizes from
  contract.blocks — untouched.
- t_invoices.tax_amount was already correct on every live invoice (0
  mismatches found against contract.tax_total before writing this batch) —
  this batch adds the missing component BREAKDOWN, it does not fix a
  wrong total.
- Discount compliance on t_invoices (subtotal/discount_total columns) is
  a deliberately separate follow-up — 0 contracts currently carry a
  discount, so nothing is broken yet, but flagged for its own review.

DELIBERATELY NOT TOUCHED (flagged, not silently skipped)
───────────────────────────────────────────────────
1. The VaNi scanner's per-event DRAFT invoice path (operations-loop/
   013_stage3_scanner_v2.sql, STEP 4) hardcodes tax_amount=0 on scanner-
   created drafts. Zero draft invoices exist in production today (verified
   live), so this is a LATENT defect, not a live undercount — but it WILL
   undercount tax the first time the scanner actually mints a draft. That
   function has been patched across 6 separate migration files; reproducing
   it here without first reading its full, multiply-patched LIVE definition
   would be a materially higher-risk edit than anything else in this batch.

2. NEW FINDING — tax registration status ("registered" vs "not GST-
   registered") has no home anywhere in the schema, and the Tax Settings
   SAVE flow (create_or_update_tax_settings / get_tax_settings_with_rates,
   called by the tax-settings edge fn) references RPC names that DO NOT
   EXIST in the live database — confirmed via direct pg_proc lookup. This
   means the tax-settings display-mode save path may currently be broken
   in production (unrelated to anything in this batch — pre-existing).
   Did NOT touch this or add a registration-status column, since there's
   no verified-safe write path to attach it to yet. Needs its own
   investigation: what actually writes t_tax_settings today (5 rows exist,
   so something does), before either the registration-status toggle or a
   fix to this save flow can be built.

3. NEW FINDING — the PUBLIC buyer review page's provider letterhead
   (logo, business name, address, phone, email — not just the new tax
   number) appears to be a live no-op. The migration file for
   validate_contract_access (contracts/010_public_contract_rpcs.sql)
   builds a full `profile` object from t_tenant_profiles, but the LIVE
   deployed function (pulled via pg_get_functiondef, verified) has no such
   fetch — its `tenant` object only returns {id, name}, no `profile` key
   at all. So contractnest-ui/src/pages/contracts/review/index.tsx has
   been reading `tenant?.profile?.*` as undefined in production, for
   fields well beyond gst_number. This batch's edit there (adding
   gst_number to providerLines) is written defensively — same
   undefined-safe pattern as the pre-existing city/phone/email lines — so
   it causes NO regression, but it also won't actually show anything until
   this separate, pre-existing RPC gap is fixed. Did not touch the RPC
   itself given the scope of the drift (more than just one field) and the
   risk in guessing what else changed between the migration file and the
   live function without a fuller audit.

Both new findings are pre-existing production issues, unrelated to this
batch's changes, surfaced while investigating where to print the tax
number. Recommend a dedicated look before Sprint 4's remaining pieces or
Sprint 7.
