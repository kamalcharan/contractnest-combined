BATCH: no-tax-save-fix
SUBMODULE(S) AFFECTED: contractnest-api, contractnest-ui, contractnest-edge

WHAT THIS BATCH IS
-------------------
Fixes saving the new "No Tax" option on Tax Settings (part of the Sprint 4
gst-tax-records work). The sprint4-gst-tax-records batch taught the RPCs and
contractnest-api's taxTypes.ts about display_mode='no_tax', but THREE
separate 2-value allow-lists elsewhere in the same request/response chain
were never updated to match, so this took two rounds to fully chase down:

Round 1 (Express middleware — caught first, from "Validation failed"):
  contractnest-api/src/validators/taxSettingsValidators.ts — request-body
  validator only allowed including_tax/excluding_tax, 400'd before the
  request ever reached the RPC.

Round 2 (found after Round 1's fix moved the error one layer deeper, to
"Invalid display_mode. Must be including_tax or excluding_tax"):
  - contractnest-edge/supabase/functions/tax-settings/index.ts — the
    deployed edge function's OWN hardcoded request check, independent of
    the RPC it calls. This is what the user actually hit after Round 1.
  - contractnest-api/src/services/taxSettingsService.ts — validateTaxSettings()
    validates the RESPONSE coming back from a successful save/fetch; would
    have rejected a successful no_tax save on its way back to the UI even
    after the edge function was fixed. Caught proactively before it could
    cause a second confusing bug report.

Round 3 (found after Round 2's fixes were deployed live — from a raw
"Cannot read properties of undefined (reading 'id')", not a validation
message, meaning the request finally reached the RPC):
  - create_or_update_tax_settings() RPC itself had a real, pre-existing bug,
    NOT specific to no_tax: `v_existing IS NOT NULL` on a PL/pgSQL RECORD
    variable checks that EVERY field is non-null, not "was a row found". Any
    tenant whose existing t_tax_settings row has a null default_tax_rate_id
    (the common case — nobody had set a specific default rate) got wrongly
    routed into the INSERT branch instead of UPDATE on ANY save, hitting the
    unique_tenant_tax_settings constraint. Fixed by using Postgres's FOUND
    special variable instead, which reliably reflects the SELECT INTO result.
    Migration: contracts/070_fix_tax_settings_upsert_record_null_check.sql.
    Applied live and verified (re-ran the exact failing call against a real
    tenant with a null default_tax_rate_id — now correctly updates in place).
  - The edge function also had a latent bug of its own: create_or_update_tax_settings
    and get_tax_settings_with_rates both catch their own exceptions internally
    and return {error: '...'} as a normal (non-throwing) response, but the
    edge function only checked the Postgres-level `error` from supabase.rpc(),
    never data?.error — so any RPC-side failure (the one above, or any future
    one) surfaced as a confusing raw TypeError instead of the real message.
    Added the missing data?.error check to both handlers.

The DB constraint and existing tenant data were otherwise already correct.

Secondary/cosmetic bug also fixed: a save failure was setting the same
state.error field the panel uses to detect a *load* failure, so any failed
save replaced the whole form with a "Failed to load tax display settings"
screen (discarding the user's in-progress selection) instead of just
toasting the error and letting them retry.

--------------------

  - display_mode .isIn([...]) allow-list now includes NO_TAX (was missing).
  - Updated the matching .withMessage() text and the VALIDATION_RULES doc
    string to mention "no_tax" too.

contractnest-api/src/services/taxSettingsService.ts
  - validateTaxSettings()'s display_mode allow-list now includes 'no_tax'.

contractnest-edge/supabase/functions/tax-settings/index.ts
  - display_mode allow-list now includes 'no_tax' in the POST handler.
  - Both handleGetRequest and handleCreateUpdateSettings now check data?.error
    (the RPC's own self-caught error shape) before assuming the payload has
    settings/rates, instead of crashing with a raw TypeError.
  - DEPLOYED LIVE already (v25) — this copy is for keeping the repo checkout
    in sync, no further deploy action needed for this file.

contractnest-edge/supabase/migrations/contracts/070_fix_tax_settings_upsert_record_null_check.sql
  - Fixes create_or_update_tax_settings()'s existence-check bug described
    above. APPLIED LIVE already — this file is for tracking only, do not
    re-run (would just no-op re-apply CREATE OR REPLACE FUNCTION, safe but
    unnecessary).

contractnest-ui/src/hooks/useTaxDisplay.ts
  - updateDisplayMode()'s catch block no longer sets state.error on a save
    failure (only saving: false) — save errors are still fully reported via
    the existing vaniToast.error('Update Failed', ...) call, this just stops
    it from also triggering the panel's full-page load-error view.
  - No change to fetchTaxSettings() — a genuine load failure still sets
    state.error and shows the "Failed to load" screen as before, correctly.
