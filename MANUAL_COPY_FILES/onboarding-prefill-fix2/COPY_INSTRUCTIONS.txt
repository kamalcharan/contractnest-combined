WHY v1 FAILED
─────────────────────────────────────────────────────
Screen 4 — ThemeSelectionStep (v1 used currentThemeId):
  currentThemeId comes from ThemeContext, which reads
  localStorage['contractnest-theme']. Every onboarding
  screen calls setTheme('vani') on mount, which overwrites
  that key to 'vani'. So by the time the user returns to
  Screen 4, currentThemeId === 'vani' — the real saved
  theme is gone from ThemeContext.

Screen 5 — IndustrySelectionStep (v1 used !servedLoading):
  useServedIndustries is enabled only when currentTenant?.id
  exists. When tenant is null at mount (auth still loading),
  the query is DISABLED — TanStack Query returns isLoading=false
  immediately with no data. !servedLoading was true right away,
  so hydration ran with an empty list and set hydrated=true.
  When the real fetch completed, hydrated=true blocked it.
  Additionally, useServedIndustriesManager does not expose
  isSuccess — only isLoading.

CORRECT FIXES
─────────────────────────────────────────────────────
Screen 4 — ThemeSelectionStep:
  Read preferred_theme from localStorage['user_data'] directly.
  user_data is written by updateUserPreferences (AuthContext)
  and is NEVER overwritten by setTheme(). It is the only
  durable source of the user's saved theme preference.
  Parse synchronously before calling setTheme('vani').

Screen 5 — IndustrySelectionStep:
  Import useServedIndustries directly alongside
  useServedIndustriesManager. Use isSuccess (true only after a
  real server response) as the hydration trigger instead of
  !isLoading (false when query is disabled or stale).
  TanStack Query deduplicates — no double fetch occurs because
  useServedIndustriesManager calls useServedIndustries internally
  with the same query key.
  Also adjusted the isLoading spinner condition to correctly
  wait until hydration is complete when a tenant is loaded.
