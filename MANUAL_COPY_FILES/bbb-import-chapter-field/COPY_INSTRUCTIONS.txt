📦 CHANGES SUMMARY
═══════════════════════════════════════════════════
Branch: bbb-import-chapter-field
(BBB going live tomorrow morning — Contacts import was mapping the
"Chapter Name" column to "Ignore" with no way to fix it)

WHAT WAS WRONG: the import mapping wizard auto-suggests a target field
per uploaded column by matching the header text against a fixed set of
regex synonyms (contactsAdapter.ts's HEADER_PATTERNS) — name, mobile,
email, business name, designation, category, tags, member id, reference.
"Chapter Name" didn't match any of them, so it always fell through to
"Ignore" — and there was no "Chapter" option in the mapping dropdown to
select manually either, because no such target field existed at all.

THE FIX: added a real "Chapter" target field, stored the same way Member
ID already is — in the contact's external_data jsonb (external_data.
chapter_name) — which the contact profile page already renders
generically (every external_data key is humanized and shown as an
"Additional Field", so "chapter_name" shows up as "Chapter Name" right
alongside "Member Id" with zero UI changes needed there). Also added
"Chapter Name" to the auto-detect patterns so it maps correctly by
default on future imports, not just this once.

Files Changed:

  contractnest-ui (1 file):
  - src/pages/contacts/import/contactsAdapter.ts
      + TargetFieldKey gains 'chapter'
      + TARGET_FIELDS gains a "Chapter" entry (dropdown option), placed
        right after "Member ID"
      + HEADER_PATTERNS gains /chapter/i, /^branch$/i, /^group$/i so
        "Chapter Name" (and similar) auto-map instead of defaulting to
        Ignore
      + ImportRow gains `chapter: string`, extracted in extractRows()
        the same way memberId already is
      + buildContactPayload() writes chapter into
        external_data.chapter_name alongside member_id when present

Submodules Affected: contractnest-ui only

Production Checklist:
  ✅ Transaction Management: N/A — this only changes what gets attached
     to the same per-row create-contact payload the import already sends;
     no new transaction boundary
  ✅ Race Condition Handling: N/A — no change to concurrency behavior
  ✅ Error Handling: unchanged — chapter is optional, same as member_id;
     no new validation/error path
  ✅ Toasts: N/A — no new user-facing action
  ✅ Loaders: N/A — no new async operation
