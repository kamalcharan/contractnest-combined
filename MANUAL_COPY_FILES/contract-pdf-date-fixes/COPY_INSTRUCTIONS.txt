Files Changed:

Submodules Affected: contractnest-ui

BUG REPORT: creating a contract via template with start date 1 Apr 2026 /
end date 31 Mar 2027 showed a preview "Service Agreement" with
"Effective Date: 21 Jul 2026" (today), "Term: 1 year -> 21 Jul 2027"
(today+1yr), and "Contract No. Draft — number on save" / "DRAFT —
AWAITING ACCEPTANCE". The actual saved contract (CN-1020) was correct in
the DB (start_date 2026-04-01, status active) — this was TWO separate
bugs in what gets displayed, not a save-time data bug.

BUG 1 — pre-save wizard preview (the one actually reported):
`ContractWizard/index.tsx`'s `case 'review':` render passes a long list
of props to `<ReviewSendStep>` but never included `startDate`, even
though `wizardState.startDate` holds the correct picked date at that
point (it's already threaded to ServiceBlocksStep and EventsPreviewStep
earlier in the same file — just dropped here). Inside ReviewSendStep,
the missing prop silently defaulted to `new Date()` (today) for both the
document preview's effective date/term math and the on-page timeline.
The "Draft — number on save" / "DRAFT — AWAITING ACCEPTANCE" markers are
CORRECT as-is — there's genuinely no contract number yet at the preview
stage, before final save.
FIX: added `startDate={wizardState.startDate}` to the ReviewSendStep call.

BUG 2 — separate, pre-existing bug affecting ALREADY-SAVED contracts:
The buyer-facing contract review/PDF page (`pages/contracts/review/index.tsx`,
reached via a CNAK magic link) and its shared renderer
(`ContractDocument.tsx`'s `buildDocFromSavedContract`) both computed the
displayed "start date" from `created_at` (when the DB row was inserted)
instead of `start_date` (the contract's actual agreed start) — for BOTH
the "Contract Duration: {date} -> {date}" range shown on the page AND the
same field baked into the exported PDF. This is independent of the
wizard bug above and would show wrong dates any time someone views/
downloads the PDF for an already-saved contract whose start date differs
from when it was created in the system.
Root cause: the backend already returns `start_date` correctly (verified
via the `get_contract_by_id` RPC and the public-validate response) — the
frontend's `FullContractData`/`SavedContractDocInput` TypeScript
interfaces simply never declared the field, so it was silently ignored
even though present in the payload.
FIX: added `start_date` to both interfaces; `buildDocFromSavedContract`
now reads `c.start_date || c.created_at` (defensive fallback for any
legacy row that somehow lacks it); split the review page's single
`startDate` variable into `createdDate` (created_at — used only for the
"· Created {date}" byline, which is correctly about record creation) and
`startDate` (now start_date — used for the "Contract Duration" range and
the PDF's effective date/term, which is correctly about the contract's
actual term).
