WHAT THIS BATCH DOES
────────────────────
Per your last two turns: the attendance policy (max no-shows, max
substitute check-ins) is now authored on the "Group Session" catalog
block itself, in Catalog Studio's Delivery Settings step — NOT a live
dashboard setting. This makes it a real, tenant-configurable T&C:

1. contractnest-ui/src/components/catalog-studio/BlockWizard/steps/service/DeliveryStep.tsx
   New "Attendance policy" card, shown only when Audience = Group,
   right after the Audience selector. Two number inputs — "Max
   no-shows allowed" / "Max substitute check-ins allowed" — both
   blank by default (= no cap). Same onChange('field', value) pattern
   already used for Cycle Period / Anchor Weekday, no new patterns.

2. contractnest-ui/src/utils/catalog-studio/catBlockAdapter.ts
   - catBlockToBlock(): exposes the saved policy as flat
     meta.maxNoShows / meta.maxSubstitutes so the wizard hydrates it
     when editing an existing block (same fallback pattern as
     serviceCycles/audience).
   - buildServiceConfig(): writes config.groupSession.attendancePolicy
     = { maxNoShows, maxSubstitutes }.
   - IMPORTANT FIX included here: buildServiceConfig previously had
     no knowledge of config.groupSession.defaultChairContactId/Name
     (set live from the Group Sessions dashboard's "Assign default
     chair", not through this wizard) — a routine Catalog Studio save
     (e.g. just changing the price) would have silently WIPED it,
     since the wizard rebuilds the whole config from scratch. Fixed:
     buildServiceConfig now always preserves whatever's already in
     config.groupSession and only overlays the policy on top. Applied
     the same preservation to the smaller partial-update path in
     blockToUpdateData() for symmetry.

WHY THIS MAKES IT A REAL T&C (not new plumbing needed)
────────────────────
Confirmed directly against real BBB data: a catalog block's full
config already gets deep-copied into custom_fields.config on EVERY
member's own t_contract_blocks row the moment their contract is
created (this is generic, already-existing, not block-type-specific —
verified against every Saturday Cadence member's contract). So the
moment a tenant sets this policy on the block, every contract created
from it afterward automatically carries the policy as part of that
member's own signed terms — no new snapshot code needed. Existing
members' contracts (created before the policy existed) simply have no
attendancePolicy in their snapshot — no cap, no false flag.
