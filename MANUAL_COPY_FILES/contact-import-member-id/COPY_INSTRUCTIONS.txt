CONTACT IMPORT — MEMBER ID (via generic external_data JSON)
Branch folder: contact-import-member-id
Submodules affected: contractnest-edge (DB — already applied/verified live; edge function fix — see note below), contractnest-ui
═══════════════════════════════════════════════════

WHAT THIS BATCH DOES
────────────────────
Prompted by a real BBB B-Hive member roster file with a "Member Id" column
(e.g. "HYDBHA2750") — an external membership number, distinct from
ContractNest's own contact ID. Previously this would silently get dumped as
unstructured text into the contact's `notes` field via the generic
"Reference Id" mapping. This batch gives it a real, structured home.

Design: a generic `external_data` JSONB column on t_contacts — NOT a
single-purpose `member_id` column — so future external-system integrations
(CRM ids, POS customer numbers, etc.) can add new keys with zero schema
change. Today it holds exactly one key: `member_id`.

Scope, per explicit direction:
  - Import-only. Not exposed on the create-contact or edit-contact forms —
    only the import wizard ever writes to it.
  - Displayed read-only on the contact profile view, in a new generic
    "External Data" card that iterates whatever keys are present — so any
    future key shows up automatically without new UI code.

⚠ REAL BUG FOUND AND FIXED VIA LIVE USER TESTING (not caught by my own
  RPC-level smoke test, which only exercised the DB layer directly):
  The database RPC (create_contact_idempotent_v2) correctly accepted and
  stored external_data from the start. But the edge function's contact
  creation code (_shared/contacts/contactService.ts, createContact method)
  builds an explicit whitelist object before calling that RPC — and it
  never included external_data, so the value was silently dropped one
  layer above the database on every real import, even though DB-level
  testing looked fully correct. Fixed by adding
  `external_data: contactData.external_data || {}` to that whitelist.
  Root-caused by tracing the full request path (UI → API → edge function →
  RPC) after live testing showed an imported contact with a real Member Id
  in the source file ending up with an empty external_data. Redeployed and
  re-verified live end-to-end after the fix.

DB (already applied live, verified against BBB data, fully reverted after
testing):
  - t_contacts.external_data jsonb DEFAULT '{}'::jsonb (new nullable column).
  - create_contact_idempotent_v2: inserts external_data from
    p_contact_data->'external_data' — the only write path, since import has
    no update-existing-contact path (confirmed: every import row is always
    a fresh create call).
  - get_contact_full_v2: returns external_data.
  - update_contact_idempotent_v2 intentionally NOT touched (import-only
    field, never sent by the edit-contact form).

Edge function (the "contacts" function — separate from the "contracts"
function touched in an earlier batch this session):
  - _shared/contacts/contactService.ts: createContact's rpcContactData
    whitelist now includes external_data.
  Diff-verified the other 6 files in this function matched the live
  deployed source byte-for-byte before redeploying (no drift). Redeployed
  live as version 65 (status ACTIVE) and re-confirmed by pulling the
  deployed source back down and finding the fix present verbatim.

Also fixed in this batch (separate ask, same conversation): when a
spreadsheet's category text doesn't fuzzy-match any known industry, the
Category → Industry mapping dropdown now defaults to the existing "Other"
industry (already in master data, id='other') instead of "— not mapped —".
Still adjustable by the user before import.

Also widened the "Member Id" header auto-detection regex to also catch
"Membership ID" / "MembershipID" / "Membership No" phrasing variants, as a
defensive robustness improvement (the BBB file's actual header, "Member
Id", already matched the original pattern).

UI (contractnest-ui):
  - pages/contacts/import/contactsAdapter.ts:
    * New `member_id` target field, split out of the old catch-all
      "Reference Id" field (which stayed generic for anything else).
    * Header auto-detection: a column named "Member Id"/"Membership
      ID"/etc. now maps to the new field instead of "Reference Id".
    * extractRows/buildContactPayload: captures the column and writes
      `external_data: { member_id: ... }` on the create payload.
  - pages/contacts/import/index.tsx: industry-fallback-to-"Other" fix
    described above (~10 lines in handleToPreview).
  - hooks/useContacts.ts: `Contact.external_data?: Record<string, string>`.
  - components/contacts/view/ContactProfileTab.tsx: new read-only
    "External Data" section (SectionCard, no edit button) below Notes —
    generic key/value viewer, only rendered when external_data has entries.
