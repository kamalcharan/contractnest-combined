BILLING/INVOICE — "ADJUSTMENT" STATUS (Phase 1 of the mid-cycle-join POA)
Branch folder: adjustment-billing-status
Submodules affected: contractnest-edge (DB + edge function, already applied/deployed live), contractnest-ui
═══════════════════════════════════════════════════

WHAT THIS BATCH DOES
────────────────────
Adds a new `adjustment` status for billing events and invoices, mirroring
the existing `bad_debt` pattern exactly — no new table. This is the
foundation for the mid-cycle-join proration workflow discussed: when a
member's fee is split (e.g. 5000 collected this year, 2500 deferred into
next year's renewal), the deferred slice gets marked `adjustment` instead
of drifting into `overdue` — distinct from `bad_debt` (gave up collecting)
because it's an intentional deferral, reconciled by the chair at renewal
via Add Note (not built yet — separate phase) and manual pricing entry.

DB (already applied live, verified against real BBB data, fully cleaned
up after testing):
  - New `adjustment` status added to m_event_status_config /
    m_event_status_transitions (global template + backfilled to all 157
    existing tenants via the same seed_event_status_defaults() mechanism
    new tenants already use — future tenants get it automatically).
  - cancel_or_writeoff_invoice: new `adjustment` action alongside
    `cancel`/`bad_debt`.
  - record_invoice_payment: rejects payment against an adjusted invoice
    (same as bad_debt/cancelled).
  - get_contract_invoices: adjustment_count in the summary.
  - run_contract_event_scanner: won't reuse an adjusted invoice as a
    target for new billing events.

⚠ TWO REAL PRE-EXISTING BUGS FOUND AND FIXED (not caused by this batch —
  they broke every existing bad_debt/cancel action too, confirmed live):
  1. cancel_or_writeoff_invoice inserted into t_contract_history columns
     that don't exist (new_status/metadata — the real columns are
     from_status/to_status/changes). Every real "Mark as Bad Debt" /
     "Cancel Invoice" click has been silently failing and rolling back.
  2. t_contract_history.action has a fixed CHECK-constraint enum that
     never included 'invoice_bad_debt' etc. — fixed to use the existing
     'status_changed' bucket, full detail still captured in `changes`.
  Both fixed as part of this migration; verified live end-to-end
  (adjustment marked, payment correctly rejected, summary count correct,
  audit row written) then fully reverted the test invoice/audit row.

Edge function (contractnest-edge/supabase/functions/contracts/index.ts,
already deployed live as version 50): widened the cancel-invoice action
validation from ['cancel','bad_debt'] to also accept 'adjustment'.
Verified the deployed function's content matches this file byte-for-byte
before deploying (no drift), and confirmed post-deploy the new action is
present.

UI (contracts/detail/index.tsx, types/contracts.ts,
operations/finance/index.tsx):
  - New "Mark as Adjustment" button next to "Mark as Bad Debt" on each
    invoice card (same seller/not-paid/not-terminal gate).
  - Confirm dialog explains the intent ("deferred, reconciled at renewal
    — not a write-off") rather than reusing the bad-debt copy.
  - Status badge/border/colors use a distinct indigo (#6366F1) so
    "Adjusted" doesn't read as an alarm color like Bad Debt/Overdue.
  - Finance view invoice-status badge: "Adjusted" label, same indigo.
  - InvoiceStatus / CancelInvoicePayload / CancelInvoiceResponse types
    widened to include 'adjustment'.

STILL TO COME (separate phases, not in this batch): Add Note on Contract
View (reusing t_audit_log), completion guard for open billing, the Renew
action (list + detail page).
