ANSWERS TO YOUR 3 QUESTIONS
────────────────────────────
1) Mid-session/per-occurrence chair change — already solved, no change
   needed. The "Change" control on a single occurrence
   (gs_schedule_assign scoped to one occurrence id) only ever touches
   that one date. Nothing in this batch affects this.

2) Future sessions not inheriting the default chair — FIXED in this
   batch. gs_schedule_assign_default now also stores the default on
   the block itself (m_cat_blocks.config.groupSession.
   defaultChairContactId / defaultChairName). gs_generate_schedule and
   gs_schedule_add now read that stored default and stamp it onto any
   NEWLY inserted occurrence row. Existing rows/overrides are untouched
   (ON CONFLICT DO NOTHING already protected them before, and still
   does). Applied live and verified: set default on Saturday Cadence,
   inserted one test date via gs_schedule_add and one via
   gs_generate_schedule — both inherited the default chair correctly;
   test rows removed afterward, real 27-row schedule and the genuine
   Sanjay Ghotikar default were left exactly as they were/should be.

3) "Master QR" — turned out to be a non-issue on the backend once you
   confirmed distinct group-session series should keep distinct QR
   codes (no collision risk). Checked the actual code: QRCard is
   already rendered with the same blockId in both the series view and
   the occurrence drill-down — meaning it was ALREADY the same one
   QR per series the whole time, resolved by date server-side
   (gs_resolve_checkin / gs_submit_checkin already do this). The only
   real problem was the QR card rendering a SECOND time on the
   occurrence drill-down page, making it look like a new QR existed
   per session date. Fixed in this batch: removed the duplicate
   <QRCard> from the occurrence view — it now only appears once, on
   the series page.
