WHAT THIS BATCH DOES
────────────────────
Phase 1 of the contact-view redesign POA. Replaces the guardrail-free
status-change path currently used by ProfileDrawer's Active/Inactive
toggle and Archive button (a bare `UPDATE t_contacts SET status = ...`
issued straight from edge-function app code — no RPC, no row lock, and
critically: once a contact hit 'archived' it could NEVER be changed again,
by anyone, because the code had a hard `IF existing.status === 'archived'
THEN throw`).

Investigated end-to-end (traced UI hook -> API -> edge function -> DB)
before building this, per owner's ask to "check the history how archive
is managed" — findings:
  - No RPC at all in that path — a bare Supabase client `.update()`.
  - Only guard: blocks ANY status change on an already-archived contact
    (so reactivation was structurally impossible, contradicting the
    intent of having an "Archived" toggle state at all).
  - Zero dependency checks — a contact with active contracts or unpaid
    invoices could be archived with no warning.
  - No real audit trail — a fire-and-forget write to the generic
    t_audit_logs table wrapped in `.catch(() => {})`; a failure there is
    invisible.
  - BUSINESS_RULES restriction banner text ("No new contracts can be
    created" for inactive contacts, etc.) is purely descriptive — nothing
    actually enforces it anywhere in the codebase.

New RPC `update_contact_status_v2`, mirroring two already-proven patterns
in this codebase (`update_contract_status`'s row-locked structured-result
shape; `delete_service_catalog_item`'s dependency-check-before-archive):
  - Row-locks the contact (`FOR UPDATE NOWAIT` — fails fast with
    CONCURRENT_UPDATE instead of hanging on contention).
  - Owner's explicit decisions (this session): archived -> active IS now
    allowed (reactivation is user-facing, not permanently blocked or
    admin-only as the old code enforced) — but archiving IS blocked
    while the contact has active contracts (status not in
    cancelled/expired/completed) or unpaid invoices (status not in
    paid/cancelled/bad_debt/adjustment), checked via the contract's
    buyer_id OR buyer_contact_person_id. Inactive has no such block —
    it's a soft pause, not terminal.
  - Idempotent no-op if already at the requested status.
  - Real audit entry written INSIDE the same transaction (not a
    fire-and-forget side call) — a failure there fails the whole
    operation instead of being silently swallowed.

VERIFIED LIVE (against real BBB data, all test writes reverted, fully
confirmed back to original state afterward):
  - Archiving a contact WITH an active contract + unpaid invoice ("D
    DASARADHI SARMA") was correctly BLOCKED:
    {"success":false,"code":"DEPENDENCY_EXISTS","error":"Cannot archive:
    1 active contract(s), 1 unpaid invoice(s)"} — contact status
    unchanged (confirmed the function's exception-block auto-rollback
    behavior works correctly too, caught via an unrelated FK-constraint
    test error, then re-verified clean).
  - Setting that SAME contact to 'inactive' succeeded (no block) —
    confirms inactive stays a soft, unrestricted pause.
  - Archiving a contact with NO active contracts/invoices ("ABHINANDAN
    DESHMUKH") succeeded.
  - Reactivating that same now-archived contact back to 'active'
    succeeded — confirms the old "archived is permanent" bug is fixed.
  - Audit rows confirmed written correctly for each transition, then
    the test-induced ones cleaned up (kept the real historical
    contact.create audit entries untouched).
