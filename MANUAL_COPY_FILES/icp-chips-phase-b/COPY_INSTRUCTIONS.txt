WHAT PHASE B DOES
  Phase A builds chips from the tenant's OWN data (profile clusters, keywords,
  resources, templates). If that yields a FULL row (4 chips), Phase B never runs.
  When it yields FEWER than 4 (cold-start / thin profile), buildSuggestions now
  borrows to fill the rest — grounded, composable starters, never generic:
    1) ICP ARCHETYPE (works from day one): classifies the tenant by industry
       (t_tenant_profiles.industry_id + t_tenant_served_industries) + persona +
       vocabulary → best m_icp_archetypes match → its chip_patterns. Falls back
       to 'generic_service' if nothing matches.
    2) EMBEDDING NEIGHBOURS (once the tenant has a profile embedding): calls
       icp_similar_tenants → borrows the strongest cluster terms of the most
       similar tenants → "1 year <Their Term> service package…".
  Precedence: own-data chips (Phase A) > neighbour > archetype > (UI static
  examples). So "same/near profiles get similar chips", and a good profile for
  one tenant improves suggestions for similar tenants.

  ENV: still needs SUPABASE_SERVICE_ROLE_KEY (the archetype table + neighbour RPC
  + profile/cluster reads are service-role). Without it, Phase A resource chips
  still work; Phase B (and vocabulary ranking) are skipped — no crash.

VERIFIED: tsc --noEmit = 0 source errors. Migration applied + verified (7
  archetypes; RPC runs — returns 0 for a tenant with no embedding, which is
  exactly when the archetype layer covers instead).
