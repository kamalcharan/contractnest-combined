Branch: sprint2-arm-catalog-studio
(Sprint 2/7 pilot — "arm catalog-studio": per-catalog-block SmartForms)

WHY THIS EXISTS: a contract doesn't pull in "the whole equipment" — it
pulls in individual catalog blocks (e.g. "Air Filter Inspection &
Replacement"), each already scoped to its own small slice of Knowledge
Tree checkpoints via `m_cat_blocks.kt_checkpoint_ids` (populated by the
existing, unmodified `ktCatBlockMapperService` at seed time — confirmed
this was already grouping checkpoints correctly, just nothing consumed
it). The single combined equipment-wide form from Step 2 covers all 97
HVAC checkpoints at once and can't be attached to any one block. This
batch adds a way to compose a small form PER catalog block and attach it
directly, so contracts inherit the right form automatically later
(no manual picking) once a block's `form_template_id` is set.

CONFIRMED LIVE (via `cat-blocks` edge function, already had this wired):
`m_cat_blocks.form_template_id` was already a real FK column to
`m_form_templates`, and the PATCH update handler already whitelisted it
(`allowedFields`) — 0 of 1731 blocks used it, but no backend change was
needed to SAVE it. Only a way to fetch blocks by equipment
(`resource_template_id` filter, added here) and a way to compose+attach
a form were missing.

Submodules Affected: contractnest-edge (live), contractnest-api, contractnest-ui

Production Checklist:
  ✅ Transaction Management: N/A for the new filter (reads only). The
     attach flow is 2 sequential calls (create form, then update block)
     — not atomic, but safe: if the update fails, the form still exists
     as a valid draft and "Regenerate" can be retried: it just creates
     another form and retries the attach, no partial/corrupt state.
  ✅ Race Condition Handling: block update already uses optimistic
     locking (version check) server-side, unchanged. Added client-side
     guard: while any block's form is being generated, ALL block buttons
     disable — the create-template mutation hook only supports one
     in-flight "create" operation at a time, so this avoids a second
     concurrent click silently no-op'ing.
  ✅ Error Handling: try/catch around the whole generate+attach flow,
     per-step toast messages (create failure vs. attach failure surfaced
     separately), matches the existing handleExport pattern in this file.
  ✅ Toasts: uses existing vaniToast.success/error — new form created,
     failed to create, failed to attach.
  ✅ Loaders: per-block spinner ("Working…") plus the existing blocksLoading
     state while the catalog block list itself loads.
