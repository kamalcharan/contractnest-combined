G1.1 — Offline UPI: config-only setup modal (add / edit / delete)
================================================================
Branch: claude/group-session-handover-reset-bsn220
Submodules Affected: contractnest-edge, contractnest-api, contractnest-ui
SUPERSEDES the "group-sessions-g1-offline-upi" bundle.

WHY
----------------------------------------------------------------
The integration setup modal was built for gateways (Test Connection →
Connect). For "Offline UPI" there is nothing to connect/test — it is plain
add / edit / delete of a stored VPA. The old modal even BLOCKED saving because
it ran a test first ("Test connection not implemented for provider:
offline_upi").

FOLLOW-UP FIX (save path) — IMPORTANT
----------------------------------------------------------------
Symptom: entered UPI details, saw a toast, but the card stayed in "Set Up"
mode and nothing persisted.
Cause: even with the modal's own test skipped, persistence goes through
useIntegrations.saveIntegration(), which ALSO runs a connection test BEFORE
the insert. For offline_upi that test returns "not implemented" -> save
aborted -> no t_tenant_integrations row -> card stays in setup mode.
Fix: saveIntegration now accepts { skipTest } and index.tsx passes
skipTest=true for config-only providers (detected via provider.metadata.
config_only), so the VPA is stored directly. Files: useIntegrations.ts +
pages/settings/integrations/index.tsx (both already in this bundle).

WHAT CHANGED
----------------------------------------------------------------
Providers flagged metadata.config_only=true (Offline UPI is) now get a
CRUD modal instead of the connect/test flow:
  - Saving NO LONGER runs a connection test (it just stores the values).
  - The "Test Connection" button is hidden.
  - Primary button reads "Save" (new) / "Save changes" (editing).
  - When a config already exists, a red "Remove" button deletes it.
This is generic — any future config-only provider gets the same treatment.

Delete was wired end-to-end (the edge function already supported it; only the
API route + hook + prop-threading were missing):
  API  : DELETE /api/integrations/:id  → integrationController.deleteIntegration
         → integrationService.deleteTenantIntegration (existing)
  UI   : useIntegrations.deleteIntegration; onDelete threaded
         index → IntegrationTypeSection → IntegrationProviderCard → modal.
