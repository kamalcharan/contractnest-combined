WHAT THIS BATCH DOES
────────────────────
Phase 2 of the contact-view redesign POA. Wires the `update_contact_status_v2`
RPC (Phase 1, already live — see contact-status-hardening batch) through
every layer, replacing the old guardrail-free bare `.update()` call, and
builds the real interactive status control in the contact-view header.

Per owner's explicit decision this session ("reactive can be user-facing,
its not a problem, onlything is, when archived - nothing is editable"):
  - The header status control stays ACTIVE/CLICKABLE at all times, including
    on an archived contact — that's the only way to reactivate it.
  - Every other edit affordance on the page is now LOCKED when
    status === 'archived': all 6 editable sections in the Profile tab
    (Identity, Contact Channels, Tags & Roles, Addresses, Compliance,
    Notes — People & Alternates was already read-only, untouched), plus
    the ActionIsland's "Profile" button (which opens ProfileDrawer, a
    second editor for the same fields — disabling it closes that
    back-door instead of leaving the lock half-enforced). The
    ActionIsland's Contract/Email/WhatsApp buttons were ALREADY disabled
    on archived contacts (pre-existing `isDisabled` logic) — no change
    needed there.

LAYER-BY-LAYER CHANGES
────────────────────
1. contractnest-edge (DEPLOYED LIVE — `contacts` edge function redeployed
   on top of v65 with only the 2 files below changed; the other 5 files
   in that function were redeployed byte-identical, verified by diffing
   against the live source before redeploying):
   - _shared/contacts/contactService.ts: `updateContactStatus()` rewritten
     to call the `update_contact_status_v2` RPC (row-locked, dependency
     checks, real audit row, allows archived->active) instead of the old
     bare `.update()` that had none of those guardrails and permanently
     blocked leaving 'archived'. On RPC failure, returns the structured
     result as-is (code/dependencies intact) instead of throwing a flat
     Error — lets the HTTP layer map it to the right status code.
   - contacts/index.ts (`handleUpdateContactStatus`): now reads
     `performed_by`/`performed_by_name` from the request, checks
     `result.success`, and maps RECORD_NOT_FOUND->404,
     DEPENDENCY_EXISTS/CONCURRENT_UPDATE->409, else 400. Removed the old
     fire-and-forget `auditLogger.log(...).catch(() => {})` call — the RPC
     already writes a real audit row inside its own transaction, so the
     old side-call was redundant and silently swallowed failures.

2. contractnest-api:
   - src/services/contactService.ts: `updateContactStatus()` signature
     extended with `userId`/`userName`, forwarded to the edge function as
     `performed_by`/`performed_by_name`. `transformForFrontend()`'s error
     branch now also passes through `dependencies` so the UI can show the
     specific "N active contract(s), M unpaid invoice(s)" count.
   - src/controllers/contactController.ts: `updateContactStatus` now reads
     `req.user?.id` / `req.user?.name || req.user?.email` and passes them
     through; failure status-code mapping expanded to include
     RECORD_NOT_FOUND (404) and DEPENDENCY_EXISTS/CONCURRENT_UPDATE (409)
     (previously only had a generic 400 path).

3. contractnest-ui:
   - hooks/useContacts.ts (`useUpdateContactStatus`): on a failed response,
     the thrown Error now carries `.code` and `.dependencies` (previously
     just a message string) so the page can distinguish a dependency block
     from any other failure.
   - pages/contacts/view.tsx:
     * Header status badge replaced with a real dropdown (Active/Inactive/
       Archived, current value checked) — click opens a small menu;
       picking a different status either calls the update directly
       (active/inactive) or opens the existing (previously unreachable)
       Archive confirmation dialog first.
     * `handleStatusUpdate` now shows a toast on both success and failure,
       surfacing the exact "Cannot archive: N active contract(s), M unpaid
       invoice(s)" message from the RPC when `err.code === 'DEPENDENCY_EXISTS'`.
     * `<ContactProfileTab>` now receives `readOnly={contact.status === 'archived'}`.
   - components/contacts/view/ContactProfileTab.tsx: new `readOnly` prop.
     Each of the 6 editable SectionCards now gets its `onEdit` handler via
     a new `editHandler()` helper that returns `undefined` when readOnly —
     this both prevents entering edit mode AND hides the section's "Edit"
     button entirely (SectionCard only renders it when `onEdit` is truthy).
   - components/contacts/cockpit/ActionIsland.tsx: the "Profile" button
     (opens ProfileDrawer — a second, overlapping editor for the same
     contact fields) now gets `disabled={isDisabled}` like the other
     buttons in this component already had (`isDisabled` = contact is
     archived) — closes the one remaining path to edit an archived
     contact's data.

FOLLOW-UP FIXES (from first round of live testing on this batch)
────────────────────
1. BUG — archiving/deactivating a contact failed with:
   "insert or update on table t_audit_logs violates foreign key
   constraint t_audit_logs_user_id_fkey"
   Root cause: contractnest-api's auth middleware
   (src/middleware/auth.ts) builds `req.user` as
   `{ id: user.id, ...response.data }` — and `response.data` (the user's
   profile row) has its own `id` field (the t_user_profiles row id, always
   a DIFFERENT uuid from the real auth id), which overwrites `id: user.id`
   in the spread. So `req.user.id`, which this batch forwarded as
   `performed_by` all the way down to the RPC's audit insert, was actually
   the profile-table id — never present in `auth.users`, hence the FK
   violation. (contact.create audits never hit this because the edge
   function derives the user id itself from the JWT, bypassing that
   layer entirely.)
   FIX (contractnest-edge/supabase/functions/contacts/index.ts only —
   deliberately NOT touching the shared auth middleware, which many other
   code paths depend on as-is): `handleUpdateContactStatus` now decodes
   the caller's own bearer token directly (new local `getAuthUserId()`
   helper, same technique `_shared/audit.ts` already uses for its own
   audit entries) instead of trusting the client-forwarded `performed_by`
   field. Redeployed live.
2. Added `contact_number` (e.g. "CT-1011") to the `get_contact_full_v2`
   RPC's response — migration
   `042_contact_number_in_full_v2.sql`, applied live. The column already
   existed and is populated via the CONTACT sequence counter on insert;
   it just wasn't in this RPC's returned JSON yet. Now shown as a small
   monospace badge next to the contact's name in the view header.
3. Removed the top-right "Edit" button in the header — per owner
   feedback it was a no-op (it only called `setActiveTab('profile')`,
   and the Profile tab — where '`activeTab`' already defaults to — is
   reachable from the tab bar anyway, with its own working, correctly
   locking Edit buttons per section). Also dropped the now-unused `Edit`
   icon import.
